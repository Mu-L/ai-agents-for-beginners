# 多代理設計模式

一旦你開始從事涉及多代理的專案，你就需要考慮多代理設計模式。然而，何時切換到多代理及其優勢可能不會立即明顯。

## 介紹

在本課程中，我們將回答以下問題：

- 多代理適用於哪些場景？
- 使用多代理相較於單一代理執行多項任務，有哪些優點？
- 實現多代理設計模式的構建模組有哪些？
- 如何了解多個代理間的互動情形？

## 學習目標

課程結束後，你應該能夠：

- 識別多代理適用的場景
- 認識使用多代理優於單一代理的優勢
- 理解實現多代理設計模式的構建模組

大局觀是什麼？

*多代理是一種設計模式，允許多個代理協作以達成共同目標*。

這種模式廣泛應用於各種領域，包括機器人技術、自主系統和分散式運算。

## 適用於多代理的場景

那麼哪些場景適合使用多代理？答案是有許多場景中使用多個代理是有益的，特別是在以下情況：

- <strong>大量工作負載</strong>：大型工作負載可細分為較小任務並分配給不同代理，允許並行處理和更快速完成。舉例來說大型資料處理任務。
- <strong>複雜任務</strong>：像大型工作負載一樣，複雜任務可分成較小子任務並分給不同專精任務特定部分的代理。例如自駕車，各代理負責導航、障礙物偵測和與其他車輛通訊。
- <strong>多元專長</strong>：不同代理擁有多元專長，讓他們比單一代理更有效率地處理任務的不同面向。舉例醫療領域，代理可負責診斷、治療計劃和病患監控。

## 多代理相較於單一代理的優勢

單一代理系統對於簡單任務可能運作良好，但對更複雜的任務，使用多代理可帶來多項優點：

- <strong>專門化</strong>：每個代理可專注於特定任務。單一代理缺乏專門化，可能面對複雜任務時會不知道該如何執行，甚至可能處理它不適合的任務。
- <strong>可擴展性</strong>：擴充系統較易新增更多代理，而非讓單一代理超負荷。
- <strong>容錯能力</strong>：若一代理失效，其他代理仍可繼續運作，確保系統可靠。

舉例來說，為用戶訂購旅程時，單一代理須處理從找航班到訂飯店和租車的所有流程。為此單一代理需具備處理所有任務的工具，可能導致系統複雜且難以維護與擴展。相較之下，多代理系統可設有專門尋找航班、訂飯店及租車的代理，使系統更模組化、易於維護且具可擴展性。

將此與由夫妻小店經營的旅行社與特許經營的旅行社相比較。夫妻小店由單一代理處理所有訂旅程事務，而特許經營則由不同代理負責各部分業務。

## 實現多代理設計模式的構建模組

在實現多代理設計模式前，你需要了解組成該模式的構建模組。

再次以為使用者訂旅程為例，構建模組包含：

- <strong>代理通訊</strong>：尋找航班、訂飯店及租車代理需溝通並分享使用者偏好與限制資訊。具體來說，尋找航班代理需與訂飯店代理溝通，確保飯店訂於航班相同日期。這意味著代理需分享使用者旅遊日期資訊，並需確定<em>哪些代理共享資訊及如何共享</em>。
- <strong>協調機制</strong>：代理需協調行動，確保符合使用者偏好與限制。使用者偏好可能是希望飯店靠近機場，限制則可能是租車僅能於機場取得，訂飯店代理需與租車代理協調，以符合這些條件，需決定<em>代理如何協調行動</em>。
- <strong>代理架構</strong>：代理需具備內部結構來決策與從與使用者互動學習。例如尋找航班代理需具備決策結構以挑選推薦航班，需決定<em>代理如何決策及從互動中學習</em>。例如，尋找航班代理可使用機器學習模型根據過去偏好推薦航班。
- <strong>多代理互動可視化</strong>：需要能查看多代理間互動情況。需有工具與技術來追蹤代理活動與互動，如記錄與監控工具、視覺化工具及效能指標。
- <strong>多代理模式</strong>：存在多種實現多代理系統的模式，如集中式、去中心化及混合架構。需選擇最適用的模式。
- <strong>人類介入</strong>：多數情況下需有人類介入，需指示代理何時要求人類介入。例如當使用者要求代理未推薦的特定飯店或航班，或需確認訂票前先詢問使用者。

## 觀看多代理間互動情況

能察看多代理間互動很重要。這種可視化對除錯、優化及確保系統整體效能絕對不可或缺。為此，需要追蹤代理活動與互動的工具與方法，如記錄與監控工具、視覺化工具及效能指標。

舉例為為使用者訂旅程，你可有一介面展示各代理狀態、使用者偏好與限制、代理間互動。介面展示使用者旅遊日期、航班代理推薦航班、飯店代理推薦飯店及租車代理推薦租車狀況。這提供清晰視圖，了解代理間互動及使用者條件是否被滿足。

接著讓我們更詳細看這些方面。

- <strong>記錄與監控工具</strong>：你應記錄代理採取的每個行動。記錄條目可儲存執行行動的代理、行動內容、行動時間及結果等資訊，可用於除錯、優化等。

- <strong>視覺化工具</strong>：視覺化工具幫助以更直覺方式觀察代理間互動。例如可用圖表展示代理間資訊流，幫助找出瓶頸、低效率和其他問題。

- <strong>效能指標</strong>：效能指標幫助追蹤多代理系統效能。例如追蹤完成任務所需時間、每單位時間完成任務數、代理推薦準確率等，協助識別改進空間並優化系統。

## 多代理模式

讓我們深入探討可用於建立多代理應用的具體模式。以下是值得考慮的一些有趣模式：

### 群組聊天

當你想建立多代理能互相通訊的群組聊天應用時，此模式很有用。典型用例包括團隊協作、客戶支援和社交網絡。

在此模式中，每個代理代表群組中的一使用者，並利用訊息協定在代理間交換訊息。代理能向群組聊天發送訊息、接收訊息並回應其他代理訊息。

此模式可用集中式架構實現，所有訊息通過中央伺服器路由，或去中心化架構，代理間直送訊息。

![群組聊天](../../../translated_images/zh-MO/multi-agent-group-chat.ec10f4cde556babd.webp)

### 任務交接

當你想建立多代理能彼此交接任務的應用時，此模式很有用。

典型用例包括客戶支援、任務管理及工作流程自動化。

在此模式中，每個代理代表一項任務或工作流程中的一步，代理可根據預設規則將任務交給其他代理。

![任務交接](../../../translated_images/zh-MO/multi-agent-hand-off.4c5fb00ba6f8750a.webp)

### 協同過濾

當你想建立多代理能合作為用戶提供推薦的應用，此模式很有用。

多代理合作的原因在於每個代理擁有不同專長，可從不同角度為推薦過程做出貢獻。

例如用戶想要股票市場最佳買股推薦：

- <strong>產業專家</strong>：一代理擁有特定產業專長。
- <strong>技術分析</strong>：另一代理擅長技術分析。
- <strong>基本面分析</strong>：還有一代理精通基本面分析。這些代理合作能為用戶提供更全面的推薦。

![推薦](../../../translated_images/zh-MO/multi-agent-filtering.d959cb129dc9f608.webp)

## 場景：退款流程

假設有一客戶嘗試退款，過程中可能涉及多個代理，讓我們把代理分為專門處理退款流程的代理和可用於其他流程的一般代理。

<strong>專門用於退款流程的代理</strong>：

以下是可能參與退款流程的一些代理：

- <strong>客戶代理</strong>：代表客戶，負責啟動退款流程。
- <strong>賣方代理</strong>：代表賣方，負責處理退款事宜。
- <strong>支付代理</strong>：代表支付過程，負責退還客戶付款。
- <strong>解決代理</strong>：代表問題解決過程，負責處理退款過程中出現的任何問題。
- <strong>合規代理</strong>：代表合規過程，負責確保退款流程符合相關法規及政策。

<strong>一般代理</strong>：

這些代理可用於業務其他部分。

- <strong>運輸代理</strong>：代表運輸過程，負責將商品寄回賣方。此代理既可用於退款流程，也可用於商品購買後的一般運輸。
- <strong>反饋代理</strong>：代表反饋過程，負責從客戶蒐集反饋。反饋可隨時取得，不僅限退款流程中。
- <strong>升級代理</strong>：代表升級過程，負責將問題升級到更高層級支援。此類代理可用於任何需要升級問題的流程。
- <strong>通知代理</strong>：代表通知過程，負責在退款流程的各階段向客戶發送通知。
- <strong>分析代理</strong>：代表分析過程，負責分析退款流程相關資料。
- <strong>審計代理</strong>：代表審計過程，負責審核退款流程以確保執行正確。
- <strong>報告代理</strong>：代表報告過程，負責生成退款流程的報告。
- <strong>知識代理</strong>：代表知識管理，負責維護退款流程相關知識庫。該代理可同時對退款及其他業務領域有所了解。
- <strong>安全代理</strong>：代表安全過程，負責確保退款流程安全。
- <strong>品質代理</strong>：代表品質管理，負責確保退款流程品質。

前述列出不少代理，既有退款流程專用者，也有可用於業務其他部分的一般代理。希望這能讓你對如何決定多代理系統中應用哪些代理有所啟發。

## 作業

設計一個用於客戶支援流程的多代理系統。識別流程中涉及的代理、其角色與責任，及代理間如何互動。請同時考慮客戶支援流程專用代理與可用於業務其他部分的一般代理。


> 在閱讀以下解決方案前先思考一下，你可能需要的代理數量比你想像的多。

> 提示：想想客戶支援流程的不同階段，並且考慮任何系統所需的代理人數。
