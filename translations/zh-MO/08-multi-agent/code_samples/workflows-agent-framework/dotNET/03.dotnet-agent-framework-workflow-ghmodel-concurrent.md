## 💻 執行程式碼

完整實作可於 `03.dotnet-agent-framework-workflow-ghmodel-concurrent.cs` 取得。該檔示範一個旅遊規劃的<strong>Fan-Out/Fan-In 併發工作流程</strong>：

### 🏗️ <strong>工作流程架構</strong>

```
User Request → ConcurrentStartExecutor → [Researcher Agent || Planner Agent] → ConcurrentAggregationExecutor → Final Output
```

**主要元件：**

1. **ConcurrentStartExecutor**：同時將用戶請求廣播給所有代理
2. **研究員代理（Researcher Agent）**：平行分析目的地與景點
3. **規劃師代理（Planner Agent）**：平行建立詳細旅行計劃
4. **ConcurrentAggregationExecutor**：彙整並合併兩個代理的結果

### 🎯 **Fan-Out/Fan-In 模式**

此工作流程展示了經典的 **Fan-Out/Fan-In** 模式：
- **Fan-Out**：一個輸入訊息同時廣播給多個代理
- <strong>併發處理</strong>：多個代理並行執行相同任務
- **Fan-In**：收集所有代理結果並合併成單一輸出

### 🔍 認識自訂執行器

**ConcurrentStartExecutor：**
- 實作 `IMessageHandler<string>` 接受字串輸入
- 將訊息廣播給所有已連接代理
- 傳送 `TurnToken` 觸發並行處理

**ConcurrentAggregationExecutor：**
- 實作 `IMessageHandler<ChatMessage>` 接收代理回應
- 以執行緒安全方式收集訊息
- 當所有預期回應到達時進行彙總
- 透過 `context.YieldOutputAsync()` 輸出最終結果

### ⚡ 效能優勢

**併發與序列比較：**
- 序列：Agent1 (30s) → Agent2 (30s) = **總共 60 秒**
- 併發：Agent1 (30s) || Agent2 (30s) = **總共 30 秒**

<strong>吞吐量提升</strong>：N 個併發代理可達到最高 N 倍速（依工作負載與資源而定）

### 🛡️ 錯誤處理

工作流程能夠優雅地處理單一代理失敗：
- 一個代理失敗，其他代理仍繼續處理
- 彙總器可實作超時邏輯
- 必要時可回傳部分結果
