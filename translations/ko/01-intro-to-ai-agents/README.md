## AI 에이전트 정의 및 AI 에이전트 유형

### AI 에이전트란 무엇인가?

간단한 설명입니다:

> **AI 에이전트는 대형 언어 모델(LLM)이 단순히 프롬프트에 반응하는 것이 아니라, 도구와 지식을 활용하여 실제 세상에서 <em>무언가를 수행</em>할 수 있게 하는 시스템입니다.**

이 내용을 조금 더 자세히 살펴보겠습니다:

- <strong>시스템</strong> — AI 에이전트는 단일 개체가 아니라 여러 구성 요소가 함께 작동하는 집합입니다. 모든 에이전트는 기본적으로 세 가지 요소로 구성됩니다:
  - <strong>환경</strong> — 에이전트가 작동하는 공간입니다. 여행 예약 에이전트라면, 예약 플랫폼 자체가 환경이 됩니다.
  - <strong>센서</strong> — 에이전트가 환경의 현재 상태를 읽는 방법입니다. 여행 에이전트는 호텔 이용 가능 여부 또는 항공권 가격을 확인할 수 있습니다.
  - <strong>행위기</strong> — 에이전트가 행동을 취하는 방법입니다. 여행 에이전트는 방을 예약하거나 확인 메시지를 보내거나 예약을 취소할 수 있습니다.

![What Are AI Agents?](../../../translated_images/ko/what-are-ai-agents.1ec8c4d548af601a.webp)

- **대형 언어 모델(LLM)** — 에이전트는 LLM 이전에도 존재했지만, 현대 에이전트를 강력하게 만드는 것은 LLM입니다. LLM은 자연어를 이해하고, 문맥을 추론하며, 모호한 사용자 요청을 구체적 실행 계획으로 변환할 수 있습니다.

- **행동 수행** — 에이전트 시스템이 없으면 LLM은 단순히 텍스트를 생성할 뿐입니다. 에이전트 시스템 안에서 LLM은 데이터베이스 검색, API 호출, 메시지 전송 등 단계적인 실행을 할 수 있습니다.

- **도구 접근권** — 에이전트가 사용할 수 있는 도구는 (1) 실행 환경과 (2) 개발자가 제공한 도구에 따라 달라집니다. 여행 에이전트는 항공편 검색은 할 수 있어도 고객 기록을 편집하지 못할 수 있습니다 — 모두 연결하는 방식에 달려 있습니다.

- **기억 + 지식** — 에이전트는 단기 기억(현재 대화)과 장기 기억(고객 데이터베이스, 과거 상호작용)을 가질 수 있습니다. 여행 에이전트는 당신이 창가 좌석을 선호한다는 것을 ‘기억’할 수도 있습니다.

---

### AI 에이전트의 다양한 유형

모든 에이전트가 동일하게 구축된 것은 아닙니다. 주요 유형을 여행 예약 에이전트를 예시로 들어 설명합니다:

| **에이전트 유형** | <strong>기능</strong> | **여행 에이전트 예시** |
|---|---|---|
| **단순 반사 에이전트** | 하드코딩된 규칙을 따름 — 기억 X, 계획 X. | 불만 이메일 발견 → 고객 서비스로 전달. 그게 전부. |
| **모델 기반 반사 에이전트** | 세상의 내부 모델을 유지하고 변화에 따라 업데이트함. | 과거 항공권 가격 기록을 추적하고 갑자기 가격이 오른 노선을 표시함. |
| **목표 기반 에이전트** | 목표를 설정하고 이를 달성하기 위한 단계별 방법을 알아냄. | 현재 위치에서 목적지까지의 전체 여행(항공편, 렌터카, 호텔) 예약. |
| **유틸리티 기반 에이전트** | 단순히 <em>하나의</em> 해결책이 아니라, 여러 요소를 따져 가장 <em>최적의</em> 해결책을 찾음. | 비용과 편의성을 균형 있게 고려해 선호도에 가장 높은 점수를 주는 여행 계획 찾기. |
| **학습 에이전트** | 피드백을 통해 시간이 지날수록 더 나아짐. | 여행 후 설문 결과를 바탕으로 미래 예약 권장사항 조정. |
| **계층적 에이전트** | 고수준 에이전트가 작업을 하위 과제로 나누어 하위 에이전트에 위임함. | ‘여행 취소’ 요청이 항공편, 호텔, 렌터카 취소로 구분되어 각각 하위 에이전트가 처리. |
| **다중 에이전트 시스템(MAS)** | 여러 독립된 에이전트가 협력하거나 경쟁함. | 협력: 호텔, 항공편, 엔터테인먼트를 각각 담당하는 에이전트들. 경쟁: 최적 가격으로 호텔 방을 채우기 위해 여러 에이전트가 경쟁. |

## AI 에이전트를 언제 사용해야 할까

AI 에이전트를 *쓸 수 있다고 해서* 항상 *써야 하는 건 아닙니다*. 에이전트가 특히 효과를 발휘하는 상황은 다음과 같습니다:

![When to use AI Agents?](../../../translated_images/ko/when-to-use-ai-agents.54becb3bed74a479.webp)

- **개방형 문제** — 문제 해결 단계가 미리 프로그래밍될 수 없을 때. LLM이 동적으로 경로를 찾아야 합니다.
- **다단계 프로세스** — 도구를 여러 단계에 걸쳐 사용해야 하고 단순 조회나 생성만으로는 부족한 작업.
- **시간에 따른 개선** — 사용자 피드백이나 환경 신호를 기반으로 시스템이 점점 똑똑해지길 원할 때.

이 과정의 후반부인 **신뢰할 수 있는 AI 에이전트 구축** 수업에서 언제 AI 에이전트를 사용할지 (그리고 사용하지 말아야 할지) 더 깊이 다룰 예정입니다.

## 에이전트형 솔루션 기초

### 에이전트 개발

에이전트를 구축할 때 가장 먼저 하는 일은 *무엇을 할 수 있는지* 정의하는 것입니다 — 도구, 행동, 행동 양식.

이 과정에서는 <strong>Microsoft Foundry Agent Service</strong>를 주 플랫폼으로 사용합니다. 이 서비스는 다음을 지원합니다:

- OpenAI, Mistral, Meta (Llama) 같은 공급자의 모델
- Tripadvisor 같은 공급자의 라이선스 데이터
- 표준화된 OpenAPI 3.0 도구 정의

### 에이전트형 패턴

LLM과 소통할 때는 프롬프트를 사용합니다. 하지만 에이전트는 모든 프롬프트를 수작업으로 작성할 수 없습니다 — 여러 단계에 걸쳐 행동해야 하기 때문입니다. 그래서 <strong>에이전트형 패턴</strong>이 필요합니다. 이는 LLM을 보다 확장 가능하고 신뢰성 있게 프롬트하고 조정하는 재사용 가능한 전략입니다.

이 과정은 가장 일반적이고 유용한 에이전트형 패턴을 중심으로 구성되어 있습니다.

### 에이전트형 프레임워크

에이전트형 프레임워크는 개발자들에게 에이전트 구축용 기성 템플릿, 도구, 인프라를 제공합니다. 이를 통해 다음 작업이 더 쉬워집니다:

- 도구 및 기능 연결하기
- 에이전트가 무엇을 하는지 관찰하고 (실패 시 디버깅하기)
- 여러 에이전트 간 협업하기

이 과정에서는 프로덕션 준비된 에이전트를 구축하기 위해 <strong>Microsoft Agent Framework (MAF)</strong>에 중점을 둡니다.
