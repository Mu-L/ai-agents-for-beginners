# بناء وكلاء استخدام الحاسوب (CUA)

يمكن لوكلاء استخدام الحاسوب التفاعل مع مواقع الويب بنفس الطريقة التي يتصرف بها الإنسان: عن طريق فتح المتصفح، فحص الصفحة، واتخاذ الإجراء الأفضل التالي بناءً على ما يرونه. في هذا الدرس، ستقوم ببناء وكيل أتمتة متصفح يبحث في Airbnb، يستخرج بيانات القوائم المنظمة، ويحدد أرخص مكان للإقامة في ستوكهولم.

يجمع الدرس بين Browser-Use للملاحة المعتمدة على الذكاء الاصطناعي، Playwright وChrome DevTools Protocol (CDP) للتحكم في المتصفح، وAzure OpenAI للمنطق المدعوم بالرؤية، وPydantic للاستخراج المنظم.

## مقدمة

سيغطي هذا الدرس:

- فهم متى يكون وكلاء استخدام الحاسوب أفضل من الأتمتة التي تعتمد فقط على API
- دمج Browser-Use مع Playwright وCDP لإدارة دورة حياة المتصفح بشكل موثوق
- استخدام رؤية Azure OpenAI ونواتج Pydantic المنظمة لاستخراج بيانات القوائم من صفحات الويب الديناميكية
- اتخاذ القرار متى تستخدم سير عمل أتمتة متصفح يعتمد على الوكيل أولاً، أو الممثل أولاً، أو مزيج بينهما

## أهداف التعلم

بعد إكمال هذا الدرس، ستعرف كيفية:

- تهيئة Browser-Use مع Azure OpenAI وPlaywright
- بناء سير عمل أتمتة متصفح يتنقل في موقع ويب حقيقي ويتعامل مع عناصر واجهة المستخدم الديناميكية
- استخراج نتائج مكتوبة من محتوى الصفحة المرئي وتحويلها إلى منطق أعمال لاحق
- الاختيار بين نمط الوكيل أو الممثل بناءً على مدى توقع مهمة المتصفح

## مثال شفري

يحتوي هذا الدرس على دفتر واحد تعليمي:

- [15-browser-user.ipynb](./15-browser-user.ipynb): يطلق جلسة Chrome عبر CDP، يبحث عن قوائم Airbnb في ستوكهولم، يستخرج الأسعار باستخدام رؤية Browser-Use، ويُرجع الخيار الأرخص كبيانات منظمة.

## المتطلبات الأساسية

- Python 3.12+
- نشر Azure OpenAI مهيأ في بيئتك
- تثبيت Chrome أو Chromium محليًا
- تثبيت تبعيات Playwright
- معرفة أساسية بـ Python غير المتزامن

## الإعداد

قم بتثبيت الحزم المستخدمة في الدفتر:

```bash
pip install browser_use playwright python-dotenv
playwright install chromium
```

قم بضبط متغيرات بيئة Azure OpenAI التي يستخدمها الدفتر:

```bash
AZURE_OPENAI_ENDPOINT=...
AZURE_OPENAI_API_KEY=...
AZURE_OPENAI_CHAT_DEPLOYMENT_NAME=...
# اختياري: الافتراضي هو أحدث إصدار من واجهة برمجة التطبيقات عند الحذف
AZURE_OPENAI_API_VERSION=...
```

## نظرة عامة على البنية

يوضح الدفتر سير عمل أتمتة متصفح هجين:

1. يبدأ Chrome مع تفعيل CDP بحيث يمكن لكل من Playwright وBrowser-Use مشاركة نفس جلسة المتصفح.
2. يتعامل وكيل Browser-Use مع مهام التنقل المفتوحة مثل فتح Airbnb، إغلاق النوافذ المنبثقة، والبحث عن ستوكهولم.
3. تُفحص الصفحة النشطة باستخدام مخطط Pydantic منظم لاستخراج عناوين القوائم، أسعار الليلة، التقييمات، وروابط URL.
4. تقارن منطق Python القوائم المستخرجة وتبرز النتيجة الأرخص.

يحتفظ هذا النهج بالمنطق المرن المدعوم بالرؤية الذي يتميز به Browser-Use، بينما يمنحك تحكمًا محددًا في المتصفح حين تحتاج إليه.

## النتائج الرئيسية وأفضل الممارسات

### متى تستخدم وكيل مقابل ممثل

| السيناريو | استخدام الوكيل | استخدام الممثل |
|----------|-----------|-----------|
| تخطيطات ديناميكية | نعم، يمكن للذكاء الاصطناعي التكيف مع تغييرات الصفحة | لا، المؤشرات الهشة يمكن أن تنهار |
| بنية معروفة | لا، الوكيل أبطأ من التحكم المباشر | نعم، سريع ودقيق |
| إيجاد العناصر | نعم، اللغة الطبيعية تعمل جيدًا | لا، تتطلب مؤشرات دقيقة |
| التحكم في التوقيت | لا، أقل توقعًا | نعم، تحكم كامل في الانتظار والمحاولات |
| سير العمل المعقد | نعم، يتعامل مع حالات واجهة المستخدم غير المتوقعة | لا، يتطلب تقسيمًا صريحًا |

### أفضل ممارسات Browser-Use

1. ابدأ بوكيل للاستكشاف والتنقل الديناميكي.
2. انتقل إلى التحكم المباشر في الصفحة عندما يصبح التفاعل متوقعًا.
3. استخدم نماذج مخرجات منظمة بحيث تكون البيانات المستخرجة موثوقة وآمنة النوع.
4. أضف تأخيرات بشكل استراتيجي بعد الإجراءات التي تثير تغييرات مرئية في واجهة المستخدم.
5. التقط لقطات شاشة أثناء التكرار لتسهيل تصحيح الأخطاء عند الفشل.
6. توقع تغييرات في مواقع الويب وصمم استراتيجيات بديلة للنوافذ المنبثقة وتحولات التخطيط.
7. امزج بين نمط الوكيل والممثل للحصول على كل من المرونة والدقة.

### ضوابط الأمان لوكلاء المتصفح

يعمل وكلاء المتصفح على مواقع ويب حية، لذلك يحتاجون إلى حدود أكثر صرامة من سكربت يستدعي فقط API معروفًا. قبل الانتقال من عرض دفتر تجريبي إلى سير عمل حقيقي، حدد الضوابط حول ما يستطيع الوكيل رؤيته، النقر عليه، وإرساله.

1. **حدد نطاق بيئة التصفح.** شغّل الوكيل في ملف تعريف متصفح مخصص أو صندوق رمل، وقيّده بالنطاقات المطلوبة فقط للمهمة.
2. **افصل المراقبة عن الفعل.** دع الوكيل يبحث، يقرأ، ويستخرج البيانات أولا؛ وطالب بخطوة موافقة صريحة قبل أن يرسل النماذج، يرسل الرسائل، يحجز سفر، يشتري، يحذف سجلات، أو يغير إعدادات الحساب.
3. **احفظ الأسرار خارج الموجهات والسجلات.** لا تضع كلمات المرور، تفاصيل الدفع، ملفات تعريف الارتباط للجلسة، أو البيانات الشخصية الخام في سياق النموذج. دع المستخدم يتولى المصادقة ويزيل الحقول الحساسة من السجلات.
4. **عامل محتوى الصفحة كمدخل غير موثوق.** قد تحتوي المواقع على تعليمات موجهة للوكيل، وليس للمستخدم. ينبغي أن يتجاهل الوكيل نص الصفحة الذي يطلب تغيير هدفه، كشف بيانات، تعطيل الحمايات، أو زيارة مواقع غير ذات صلة.
5. **استخدم التحقق الحتمي حول الخطوات الخطرة.** تحقق من عنوان URL الحالي، عنوان الصفحة، العنصر المختار، السعر، المستلم، وملخص الإجراء بالشفرة قبل طلب موافقة المستخدم على الخطوة النهائية.
6. **حدد الميزانيات وشروط الإيقاف.** حدد عدد الإجراءات، المحاولات، الألسنة، والدقائق التي يمكن للوكيل استخدامها. توقف عندما تكون حالة الصفحة غامضة بدلاً من الاستمرار في النقر.
7. **سجل الأدلة المفيدة، وليس الكل.** احتفظ بملخصات الإجراءات، الطوابع الزمنية، عناوين URL، أوصاف العناصر المختارة، ومرجع لقطات الشاشة بحيث يمكن مراجعة الفشل دون تخزين محتوى الصفحة الحساسة غير الضروري.

في مثال Airbnb، الخيار الآمن الافتراضي هو البحث في القوائم واستخراج الأسعار. يجب أن يكون تسجيل الدخول، الاتصال بالمضيف، أو إكمال الحجز إجراءً منفصلًا بموافقة المستخدم.

### تطبيقات في العالم الحقيقي

- حجز السفر ومراقبة الأسعار
- مقارنة الأسعار في التجارة الإلكترونية وفحص التوافر
- استخراج منظم من مواقع ويب ديناميكية
- اختبار واجهات المستخدم والتحقق منها باستخدام الرؤية الحاسوبية
- مراقبة المواقع والتنبيه
- ملء النماذج الذكي عبر التدفقات متعددة الخطوات

## مثال واقعي: مشروع أوبال من مايكروسوفت

الوكيل الذي تبنيه في هذا الدرس هو نسخة صغيرة ومحلية من **وكيل استخدام الحاسوب (CUA)** — برنامج يدير المتصفح كما لو كان شخصًا. تقدم مايكروسوفت نفس الفكرة للمؤسسات من خلال **[مشروع أوبال (Frontier)](https://support.microsoft.com/en-us/microsoft-365-copilot/get-started-with-project-opal-frontier)**، ميزة في Microsoft 365 Copilot.

مع مشروع أوبال، تصف مهمة ويعمل الوكيل نيابة عنك باستخدام **استخدام الحاسوب على جهاز سحابي Windows 365 آمن**، يعمل عبر تطبيقات المتصفح، المواقع، وبيانات مؤسستك. يعمل **بشكل غير متزامن في الخلفية**، ويمكنك توجيه العمل أو السيطرة عليه في أي وقت. تشمل الوظائف المثالية:

- إدارة طلبات عضوية مجموعات الأمان
- جمع والتحقق من أدلة التدقيق لمراجعات الالتزام
- معالجة حوادث تكنولوجيا المعلومات (تحديث حالة التذاكر، تعيين المسؤولين، إغلاق التكرارات)
- تجميع بيانات Excel في عرض إغلاق مالي

أوبال هو مرجع مفيد لما يبدو عليه وكيل استخدام الحاسوب **ذو جودة إنتاج وجدير بالثقة** — ويعزز مفاهيم من الدروس السابقة:

| المفهوم في هذا الدورة | كيف يطبق مشروع أوبال ذلك |
|------------------------|-----------------------------|
| **الإنسان في الحلقة** (الدرس 06) | يتوقف أوبال لبيانات تسجيل الدخول، البيانات الحساسة، أو التعليمات الغامضة، ولا يدخل كلمات السر أو يرسل النماذج بدون تأكيد صريح. يمكنك *السيطرة* و*إعادة السيطرة* في منتصف المهمة. |
| **وكلاء موثوقون وآمنون** (الدرس 06 و18) | يعمل في جهاز Windows 365 السحابي المعزول، متصفح فقط افتراضيًا (يتم حظر الوصول إلى الحاسوب الآخر، مطبق عبر Intune)، يستخدم هويتك للوصول فقط إلى ما تُصرح به، ويسجل كل إجراء للمراجعة. |
| **التخطيط والمعرفة العليا** (الدرس 07 و09) | ينتج أوبال خطة للعمل أولاً، ثم يشرف على تفكيره في كل خطوة ويتوقف إذا اكتشف نشاطًا مريبًا. |
| **القدرات / الأدوات القابلة لإعادة الاستخدام** (الدرس 04) | تتيح **المهارات** كتابة تعليمات للمهام المتكررة (مستوردة من ملف `.md` أو مكتوبة مع أوبال) وإعادة استخدامها عبر المحادثات. |

> **التوفر:** مشروع أوبال متوفر حاليًا للمستخدمين في [برنامج الوصول المبكر Frontier](https://adoption.microsoft.com/copilot/frontier-program/) مع اشتراك Microsoft 365 Copilot، ويجب على المسؤول إكمال الإعداد. ولأنه ميزة تجريبية في Frontier، قد تتغير القدرات مع الوقت.
