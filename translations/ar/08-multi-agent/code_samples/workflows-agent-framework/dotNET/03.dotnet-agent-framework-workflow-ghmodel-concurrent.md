# ⚡ سير العمل المتزامن للوكيل مع Azure OpenAI (Responses API) (.NET)

## 📋 برنامج تعليمي للمعالجة المتوازية عالية الأداء

يظهر هذا الدفتر **أنماط سير العمل المتزامنة** باستخدام إطار عمل الوكيل من مايكروسوفت لـ .NET و Azure OpenAI (Responses API). سوف تتعلم كيفية بناء سير عمل عالي الأداء ومتوازي يعزز الإنتاجية من خلال تشغيل عدة وكلاء ذكاء اصطناعي في نفس الوقت مع الحفاظ على التنسيق وتناسق البيانات.

## 🎯 أهداف التعلم

### 🚀 **أساسيات المعالجة المتزامنة**
- **تشغيل الوكلاء بشكل متوازي**: تشغيل عدة وكلاء ذكاء اصطناعي في نفس الوقت لأقصى أداء
- **نموذج Async/Await**: الاستفادة من نموذج البرمجة غير المتزامنة في .NET للفعالية في التزامن
- **Azure OpenAI (Responses API)**: تنسيق استدعاءات متزامنة متعددة إلى Azure OpenAI Responses API
- **إدارة الموارد**: إدارة موارد نموذج الذكاء الاصطناعي بفعالية عبر العمليات المتزامنة

### 🏗️ **الهيكلية المتقدمة للتزامن**
- **الموازاة المعتمدة على المهام**: استخدام مكتبة المهام المتوازية في .NET لأداء متزامن مثالي
- **أنماط المزامنة**: تنسيق الوكلاء المتزامنين مع تجنب مشاكل التسابق
- **توازن التحميل**: توزيع العمل بكفاءة عبر قدرة المعالجة المتزامنة المتوفرة
- **تحمل الأخطاء**: التعامل مع فشل الوكيل الفردي دون إيقاف سير العمل بأكمله

### 🏢 **تطبيقات المؤسسات المتزامنة**
- **معالجة الوثائق ذات الحجم الكبير**: معالجة عدة وثائق في نفس الوقت
- **تحليل المحتوى في الوقت الحقيقي**: تحليل متزامن لتدفقات البيانات الواردة
- **تحسين المعالجة التراكمية**: زيادة الإنتاجية لعمليات معالجة البيانات واسعة النطاق

## 💻 تشغيل الكود

التنفيذ الكامل متاح في `03.dotnet-agent-framework-workflow-ghmodel-concurrent.cs`. يُظهر هذا الملف **سير عمل Fan-Out/Fan-In المتزامن** لتخطيط السفر:

### 🏗️ **هيكلية سير العمل**

```
User Request → ConcurrentStartExecutor → [Researcher Agent || Planner Agent] → ConcurrentAggregationExecutor → Final Output
```

**المكونات الرئيسية:**

1. **ConcurrentStartExecutor**: يوزع طلب المستخدم إلى كل الوكلاء في نفس الوقت
2. **وكيل الباحث**: يحلل الوجهات والمعالم بشكل متزامن
3. **وكيل المخطط**: ينشئ خطط سفر مفصلة بالتوازي
4. **ConcurrentAggregationExecutor**: يجمع ويدمج النتائج من كلا الوكيلين

### 🎯 **نمط Fan-Out/Fan-In**

يُظهر هذا سير العمل النمط الكلاسيكي **Fan-Out/Fan-In**:
- **Fan-Out**: رسالة إدخال واحدة تُبث إلى عدة وكلاء في الوقت نفسه
- **المعالجة المتزامنة**: يعمل عدة وكلاء بالتوازي على نفس المهمة
- **Fan-In**: تُجمع النتائج من جميع الوكلاء وتُدمج في مخرج واحد

### 📝 المخرجات المتوقعة

سير العمل سيقوم بـ:
1. **إرسال الطلب**: إرسال "خطة رحلة إلى سياتل في ديسمبر" إلى كلا الوكلاء
2. **المعالجة المتزامنة**: يعمل الوكلاء معًا في نفس الوقت:
   - الباحث يحدد المعالم والتفاصيل
   - المخطط ينشئ الجدول اللوجستي وخطة الرحلة
3. **التجميع**: دمج كل الاستجابات في مخرج شامل
4. **عرض النتائج**: عرض خطة السفر المدمجة بكل المعلومات

### 🔍 فهم المشغلات المخصصة

**ConcurrentStartExecutor:**
- يطبق `IMessageHandler<string>` لقبول مدخلات نصية
- يبث الرسائل إلى جميع الوكلاء المتصلين
- يرسل `TurnToken` لتحفيز المعالجة المتزامنة

**ConcurrentAggregationExecutor:**
- يطبق `IMessageHandler<ChatMessage>` لاستلام استجابات الوكيل
- يجمع الرسائل بطريقة آمنة حسب الخيوط
- يجري التجميع عندما تصل كل الاستجابات المتوقعة
- يعطي المخرجات النهائية باستخدام `context.YieldOutputAsync()`

### ⚡ فوائد الأداء

**المعالجة المتزامنة مقابل المتتالية:**
- المتتالية: وكيل1 (30ث) → وكيل2 (30ث) = **60 ثانية إجمالي**
- المتزامنة: وكيل1 (30ث) || وكيل2 (30ث) = **30 ثانية إجمالي**

**تحسين الإنتاجية**: أسرع حتى N مرة لـ N وكلاء متزامنين (يعتمد على عبء العمل والموارد)
