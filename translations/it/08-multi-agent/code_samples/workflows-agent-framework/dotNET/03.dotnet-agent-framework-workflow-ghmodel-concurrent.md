## 💻 Esecuzione del Codice

L’implementazione completa è disponibile in `03.dotnet-agent-framework-workflow-ghmodel-concurrent.cs`. Questo file dimostra un **flusso di lavoro concorrente Fan-Out/Fan-In** per la pianificazione viaggi:

### 🏗️ **Architettura del Flusso di Lavoro**

```
User Request → ConcurrentStartExecutor → [Researcher Agent || Planner Agent] → ConcurrentAggregationExecutor → Final Output
```

**Componenti Chiave:**

1. **ConcurrentStartExecutor**: Trasmette la richiesta utente simultaneamente a tutti gli agenti
2. **Agente Ricercatore**: Analizza destinazioni e attrazioni in parallelo
3. **Agente Pianificatore**: Crea piani di viaggio dettagliati in parallelo
4. **ConcurrentAggregationExecutor**: Raccoglie e fonde i risultati di entrambi gli agenti

### 🎯 **Pattern Fan-Out/Fan-In**

Questo flusso di lavoro illustra il classico pattern **Fan-Out/Fan-In**:
- **Fan-Out**: Un messaggio di input viene trasmesso simultaneamente a più agenti
- **Elaborazione Concorrente**: Molti agenti lavorano in parallelo sulla stessa attività
- **Fan-In**: I risultati di tutti gli agenti sono raccolti e aggregati in un unico output

Il flusso di lavoro:

### 🔍 Comprendere gli Executor Personalizzati

**ConcurrentStartExecutor:**
- Implementa `IMessageHandler<string>` per accettare input stringa
- Trasmette messaggi a tutti gli agenti connessi
- Invia `TurnToken` per attivare l’elaborazione concorrente

**ConcurrentAggregationExecutor:**
- Implementa `IMessageHandler<ChatMessage>` per ricevere risposte dagli agenti
- Raccoglie i messaggi in modo thread-safe
- Aggrega quando arrivano tutte le risposte previste
- Produce l’output finale usando `context.YieldOutputAsync()`

### ⚡ Benefici Prestazionali

**Concorrente vs Sequenziale:**
- Sequenziale: Agente1 (30s) → Agente2 (30s) = **60 secondi totali**
- Concorrente: Agente1 (30s) || Agente2 (30s) = **30 secondi totali**

**Miglioramento del throughput**: Fino a N× più veloce per N agenti concorrenti (a seconda di carico e risorse)

### 🛡️ Gestione degli Errori

Il flusso di lavoro gestisce con grazia i fallimenti degli agenti individuali:
- Se un agente fallisce, gli altri continuano a elaborare
- L’aggregatore può implementare logica di timeout
- Possono essere restituiti risultati parziali se necessario
