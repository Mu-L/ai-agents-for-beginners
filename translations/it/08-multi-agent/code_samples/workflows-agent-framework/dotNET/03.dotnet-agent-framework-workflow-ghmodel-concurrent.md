## 💻 Esecuzione del Codice

L'implementazione completa è disponibile in `03.dotnet-agent-framework-workflow-ghmodel-concurrent.cs`. Questo file dimostra un **flusso di lavoro concorrente Fan-Out/Fan-In** per la pianificazione di viaggi:

### 🏗️ **Architettura del Flusso di Lavoro**

```
User Request → ConcurrentStartExecutor → [Researcher Agent || Planner Agent] → ConcurrentAggregationExecutor → Final Output
```

**Componenti Chiave:**

1. **ConcurrentStartExecutor**: Trasmette la richiesta dell’utente a tutti gli agenti simultaneamente
2. **Agente Ricercatore**: Analizza destinazioni e attrazioni in modo concorrente
3. **Agente Pianificatore**: Crea piani di viaggio dettagliati in parallelo
4. **ConcurrentAggregationExecutor**: Raccoglie e unisce i risultati di entrambi gli agenti

### 🎯 **Modello Fan-Out/Fan-In**

Questo flusso di lavoro dimostra il classico modello **Fan-Out/Fan-In**:
- **Fan-Out**: Un messaggio di input viene trasmesso a più agenti simultaneamente
- **Elaborazione Concorrente**: Più agenti lavorano in parallelo sullo stesso compito
- **Fan-In**: I risultati di tutti gli agenti sono raccolti e aggregati in un’unica uscita

### 🔍 Comprendere gli Esecutori Personalizzati

**ConcurrentStartExecutor:**
- Implementa `IMessageHandler<string>` per accettare input di tipo stringa
- Trasmette messaggi a tutti gli agenti connessi
- Invia `TurnToken` per avviare il processamento concorrente

**ConcurrentAggregationExecutor:**
- Implementa `IMessageHandler<ChatMessage>` per ricevere risposte dagli agenti
- Raccoglie messaggi in modo thread-safe
- Esegue l’aggregazione quando arrivano tutte le risposte attese
- Produce l’output finale usando `context.YieldOutputAsync()`

### ⚡ Benefici in Termini di Prestazioni

**Concorrente vs Sequenziale:**
- Sequenziale: Agente1 (30s) → Agente2 (30s) = **60 secondi totali**
- Concorrente: Agente1 (30s) || Agente2 (30s) = **30 secondi totali**

**Miglioramento del throughput**: Fino a N× più veloce per N agenti concorrenti (dipende da carico e risorse)

### 🛡️ Gestione degli Errori

Il flusso di lavoro gestisce i fallimenti di singoli agenti in modo fluido:
- Se un agente fallisce, gli altri continuano l’elaborazione
- L’aggregatore può implementare logica di timeout
- Risultati parziali possono essere restituiti se necessario
