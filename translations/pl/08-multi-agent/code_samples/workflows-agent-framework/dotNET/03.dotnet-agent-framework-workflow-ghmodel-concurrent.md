# ⚡ Równoległe Przepływy Pracy Agentów z Azure OpenAI (Responses API) (.NET)

## 📋 Samouczek Wysokowydajnego Przetwarzania Równoległego

Ten notatnik demonstruje **wzorce równoległych przepływów pracy** wykorzystując Microsoft Agent Framework dla .NET i Azure OpenAI (Responses API). Nauczysz się, jak budować wysokowydajne, równoległe przepływy pracy, które maksymalizują przepustowość przez jednoczesne uruchamianie wielu agentów AI, utrzymując przy tym koordynację i spójność danych.

## 🎯 Cele Nauki

### 🚀 **Podstawy Przetwarzania Równoległego**
- **Równoczesne Uruchamianie Agentów**: Uruchamiaj wielu agentów AI jednocześnie dla maksymalnej wydajności
- **Wzorce Async/Await**: Wykorzystaj model programowania asynchronicznego .NET do efektywnej współbieżności
- **Azure OpenAI (Responses API)**: Koordynuj wiele równoczesnych wywołań do Azure OpenAI Responses API
- **Zarządzanie Zasobami**: Efektywnie zarządzaj zasobami modeli AI podczas równoległych operacji

### 🏗️ **Zaawansowana Architektura Współbieżności**
- **Równoległość oparta na Zadaniach**: Wykorzystaj Task Parallel Library .NET do optymalnego wykonywania równoległego
- **Wzorce Synchronizacji**: Koordynuj równoczesnych agentów unikając warunków wyścigu
- **Równoważenie Obciążenia**: Efektywnie rozdziel zadania na dostępną moc przetwarzania równoległego

### 🏢 **Równoległe Aplikacje Przedsiębiorstw**

## 💻 Uruchamianie Kodów

Pełna implementacja dostępna jest w pliku `03.dotnet-agent-framework-workflow-ghmodel-concurrent.cs`. Ten plik demonstruje **równoległy przepływ pracy Fan-Out/Fan-In** dla planowania podróży:

### 🏗️ **Architektura Przepływu Pracy**

```
User Request → ConcurrentStartExecutor → [Researcher Agent || Planner Agent] → ConcurrentAggregationExecutor → Final Output
```

**Kluczowe Komponenty:**

1. **ConcurrentStartExecutor**: Nadaje równocześnie żądanie użytkownika do wszystkich agentów
2. **Agent Badacz**: Równocześnie analizuje cele podróży i atrakcje
3. **Agent Planista**: Tworzy szczegółowe plany podróży równolegle
4. **ConcurrentAggregationExecutor**: Zbiera i scala wyniki obu agentów

### 🎯 **Wzorzec Fan-Out/Fan-In**

Ten przepływ pracy demonstruje klasyczny wzorzec **Fan-Out/Fan-In**:
- **Fan-Out**: Jeden komunikat wejściowy jest nadawany do wielu agentów jednocześnie
- **Przetwarzanie równoległe**: Wielu agentów pracuje jednocześnie nad tym samym zadaniem
- **Fan-In**: Wyniki ze wszystkich agentów są zbierane i agregowane do pojedynczego wyniku

### 🔍 Zrozumienie Własnych Executorów

**ConcurrentStartExecutor:**
- Implementuje `IMessageHandler<string>` do akceptowania wejścia typu string
- Nadaje komunikaty do wszystkich podłączonych agentów
- Wysyła `TurnToken`, aby wywołać przetwarzanie równoległe

**ConcurrentAggregationExecutor:**
- Implementuje `IMessageHandler<ChatMessage>`, aby odbierać odpowiedzi agentów
- Zbiera komunikaty w sposób bezpieczny dla wątków
- Agreguje, gdy nadejdą wszystkie oczekiwane odpowiedzi
- Zwraca ostateczny wynik przy użyciu `context.YieldOutputAsync()`

### ⚡ Korzyści Wydajnościowe

**Równoległe vs Sekwencyjne:**
- Sekwencyjne: Agent1 (30s) → Agent2 (30s) = **60 sekund łącznie**
- Równoległe: Agent1 (30s) || Agent2 (30s) = **30 sekund łącznie**

**Poprawa przepustowości**: Nawet N razy szybsze dla N równoległych agentów (zależnie od obciążenia i zasobów)

### 🛡️ Obsługa Błędów

Przepływ pracy łagodnie obsługuje awarie pojedynczych agentów:
- Jeśli jeden agent zawiedzie, pozostałe kontynuują przetwarzanie
- Agregator może implementować logikę limitu czasu
- Częściowe wyniki mogą być zwracane w razie potrzeby
