### 🏗️ **រចនាសម្ព័ន្ធ Concurrency ជាមួយជំហានកំពូល**

- **Load Balancing**: ចែកចាយការងារយ៉ាងមានប្រសិទ្ធភាពនៅលើសមត្ថភាពដំណើរការ concurrency
- **ភាពអាចសម្ពាធបាន Fault**: ដោះស្រាយបញ្ហាខូចខាតរបស់អ្នកប្រើជាលក្ខណៈផ្ទាល់ខ្លួនដោយមិនឈប់ដំណើរការទាំងមូល

## 💻 ការរត់កូដ

ការអនុវត្តទាំងមូលមានក្នុង `03.dotnet-agent-framework-workflow-ghmodel-concurrent.cs`។ ហើយឯកសារនេះបង្ហាញពី **Fan-Out/Fan-In workflow concurrent** សម្រាប់ផែនការធ្វើដំណើរ៖

### 🏗️ **រចនាសម្ព័ន្ធ Workflow**

```
User Request → ConcurrentStartExecutor → [Researcher Agent || Planner Agent] → ConcurrentAggregationExecutor → Final Output
```

**ធាតុសំខាន់:**

1. **ConcurrentStartExecutor**: បញ្ជូនសំណើរអ្នកប្រើទៅអាជ្ញាធរទាំងអស់ក្នុងពេលតែមួយ
2. **Researcher Agent**: វិភាគទីកន្លែង និងទេសភាពដោយ parallel
3. **Planner Agent**: បង្កើតផែនការធ្វើដំណើរយ៉ាងលម្អិតដោយ parallel
4. **ConcurrentAggregationExecutor**: ប្រមូលនិងគំរើសលទ្ធផលពីអាជ្ញាធរទាំងពីរ

### 🎯 **គំរូ Fan-Out/Fan-In**

Workflow នេះបង្ហាញគំរូ **Fan-Out/Fan-In** ដ៏ល្បី៖
- **Fan-Out**: សារចូលមួយត្រូវបានផ្សាយទៅអាជ្ញាធរច្រើននៅពេលតែមួយ
- **ដំណើរការបន្តបន្ទាប់**: អ្នកជំនាញច្រើនដំណើរការនៅពេលតែមួយលើភារកិច្ចដូចគ្នា
- **Fan-In**: លទ្ធផលពីអ្នកជំនាញទាំងអស់ត្រូវបានប្រមូល និងបង្រួមទៅជា result តែមួយ

### 🎯 កម្មវិធីពិភពលោកពិតប្រាកដ

គំរូ concurrent នេះឆរឆាយសម្រាប់:
- **បង្កើតមាតិកា**: អ្នកនិពន្ធច្រើនបង្កើតផ្នែកខុសៗគ្នានៅពេលតែមួយ
- **ពិនិត្យកូដ**: អ្នកពិនិត្យច្រើនវិភាគកូដពីទស្សនៈផ្សេងៗ
- **ស្រាវជ្រាវទីផ្សារ**: វិភាគសម្រង់ទីផ្សារផ្សេងៗដោយ parallel
- **ដំណើរការឯកសារ**: ការដកស្រង់, វិភាគ និងបញ្ជាក់ដោយ concurrent
- **វិភាគមើលពីចំណុចមើលច្រើន**: ទទួលបានទស្សនៈផ្សេងៗលើសារជាក់ស្តែង

### 🔍 ការយល់ដឹងអំពី Custom Executors

**ConcurrentStartExecutor:**
- អនុវត្ត `IMessageHandler<string>` សម្រាប់ទទួលសារប្រភេទ string
- ផ្សាយសារទៅអ្នកជំនាញទាំងអស់ដែលភ្ជាប់
- ផ្ញើ `TurnToken` ដើម្បីចាប់ផ្តើមប្រតិបត្តិការបន្តបន្ទាប់

**ConcurrentAggregationExecutor:**
- អនុវត្ត `IMessageHandler<ChatMessage>` ដើម្បីទទួលចម្លើយពីអ្នកជំនាញ
- ប្រមូលសារជាផ្នែក thread-safe
- បង្រួមពេលចម្លើយទាំងអស់ដែលរំពឹងទុកបានមកដល់
- ផ្ដល់លទ្ធផលចុងក្រោយដោយប្រើ `context.YieldOutputAsync()`

### ⚡ អត្ថប្រយោជន៍ប្រសិទ្ធភាព

**Concurrent ទល់នឹង Sequential:**
- Sequential: Agent1 (30s) → Agent2 (30s) = **60 វិនាទីសរុប**
- Concurrent: Agent1 (30s) || Agent2 (30s) = **30 វិនាទីសរុប**

**ការកែលម្អ throughput**: លឿនច្រើនដងសម្រាប់ N អ្នកជំនាញ concurrency (អាស្រ័យលើភារកិច្ចនិងធនធាន)

### 🛡️ ការដោះស្រាយកំហុស

Workflow នេះដោះស្រាយកំហុសអ្នកជំនាញបុគ្គលដោយការរស់រានមានលក្ខណៈល្អ៖
- ប្រសិនបើអ្នកជំនាញមួយបរាជ័យ អ្នកជំនាញដទៃទៀតនៅតែបន្តដំណើរការ
- អ្នកបង្រួមអាចអនុវត្តដំណាក់កាល timeout
- លទ្ធផលផ្នែកអាចត្រូវបានត្រឡប់មកវិញបើចាំបាច់
