## 💻 Spuštění kódu

Kompletní implementace je k dispozici v `03.dotnet-agent-framework-workflow-ghmodel-concurrent.cs`. Tento soubor demonstruje **Fan-Out/Fan-In souběžný pracovní postup** pro plánování cest:

### 🏗️ **Architektura pracovního postupu**

```
User Request → ConcurrentStartExecutor → [Researcher Agent || Planner Agent] → ConcurrentAggregationExecutor → Final Output
```

**Klíčové komponenty:**

1. **ConcurrentStartExecutor**: Vysílá uživatelskou žádost ke všem agentům současně
2. **Výzkumný agent**: Současně analyzuje cíle a atrakce
3. **Plánovací agent**: Současně vytváří podrobné cestovní plány
4. **ConcurrentAggregationExecutor**: Sbírá a slučuje výsledky od obou agentů

### 🎯 **Fan-Out/Fan-In vzor**

Tento pracovní postup demonstruje klasický **Fan-Out/Fan-In** vzor:
- **Fan-Out**: Jeden vstupní signál je vysílán současně více agentům
- **Souběžné zpracování**: Více agentů pracuje paralelně na stejném úkolu
- **Fan-In**: Výsledky všech agentů jsou sbírány a agregovány do jednoho výstupu

### 🔍 Pochopení vlastních executorů

**ConcurrentStartExecutor:**
- Implementuje `IMessageHandler<string>` pro příjem textového vstupu
- Vysílá zprávy všem připojeným agentům
- Posílá `TurnToken` k zahájení souběžného zpracování

**ConcurrentAggregationExecutor:**
- Implementuje `IMessageHandler<ChatMessage>` pro přijímání odpovědí agentů
- Sbírá zprávy bezpečně pro vícenásobné vlákna
- Agreguje, když přijdou všechny očekávané odpovědi
- Vydává konečný výstup pomocí `context.YieldOutputAsync()`

### ⚡ Přínosy pro výkon

**Souběžné vs Sekvenční:**
- Sekvenční: Agent1 (30s) → Agent2 (30s) = **60 sekund celkem**
- Souběžné: Agent1 (30s) || Agent2 (30s) = **30 sekund celkem**

**Zlepšení propustnosti**: Až N× rychlejší pro N souběžných agentů (v závislosti na zátěži a zdrojích)

### 🛡️ Zpracování chyb

Pracovní postup elegantně řeší selhání jednotlivých agentů:
- Pokud jeden agent selže, ostatní pokračují v zpracování
- Agregátor může implementovat logiku timeoutu
- V případě potřeby lze vracet částečné výsledky
