## AIエージェントの定義とタイプ

### AIエージェントとは？

簡単に考える方法はこちらです：

> **AIエージェントは、大規模言語モデル（LLM）に単なる応答ではなく、ツールと知識を与えて実際に<em>行動させる</em>システムです。**

少し詳しく見ていきましょう：

- <strong>システム</strong> — AIエージェントは単一のものではありません。複数の部品が連携して動く集合体です。基本的にエージェントは次の3つの要素を持ちます：
  - <strong>環境</strong> — エージェントが動く領域。旅行予約エージェントなら、予約プラットフォームそのものです。
  - <strong>センサー</strong> — 環境の現在の状態を読み取る方法。旅行エージェントはホテルの空き状況や航空券の価格を確認します。
  - <strong>アクチュエーター</strong> — エージェントが行動を起こす方法。予約、確認メールの送信、キャンセルなどを行います。

![AIエージェントとは？](../../../translated_images/ja/what-are-ai-agents.1ec8c4d548af601a.webp)

- <strong>大規模言語モデル</strong> — エージェントはLLMが登場する前からありましたが、LLMが現代のエージェントを強力にしています。自然言語の理解、文脈の推論、曖昧な要求を具体的な行動計画に変換できます。

- <strong>行動の実行</strong> — エージェントシステムなしではLLMは単に文章を生成するだけです。システム内では実際に手順を実行できます。データベース検索、API呼び出し、メッセージ送信などです。

- <strong>ツールへのアクセス</strong> — 使えるツールは、(1) エージェントが動作する環境と (2) 開発者が与えたものによります。旅行エージェントは航空便の検索はできても顧客情報の編集はできないかもしれません。全ては配線次第です。

- **メモリー＋知識** — 短期メモリー（現在の会話）と長期メモリー（顧客データベース、過去のやり取り）を持つことができます。旅行エージェントは「窓際席が好み」と覚えているかもしれません。

### AIエージェントの種類

全てのエージェントが同じ作りではありません。ここで主要なタイプを、旅行予約エージェントを例に分解します：

| <strong>エージェントタイプ</strong> | <strong>機能</strong> | <strong>旅行代理店例</strong> |
|---|---|---|
| <strong>単純反射型エージェント</strong> | ハードコードされたルールに従う — メモリーなし、計画なし。 | 苦情メールを見て → カスタマーサービスに転送。ただそれだけ。 |
| <strong>モデルベース反射型エージェント</strong> | 内部モデルを持ち、変化に応じて更新。 | 過去のフライト価格を追跡し、急に高くなった区間を知らせる。 |
| <strong>目標志向エージェント</strong> | 目標を持ち、それに向けて段階的に方法を決定。 | 出発地から目的地までのフライト、車、ホテルの予約をまとめて行う。 |
| <strong>効用ベースエージェント</strong> | 妥協点を比較し、<em>最良</em>の解を見つける。 | 費用と利便性のバランスを取り、好みに最適な旅行を提案。 |
| <strong>学習エージェント</strong> | フィードバックを元に時間をかけて改善する。 | 旅行後のアンケート結果に基づいて、次回の予約提案を調整。 |
| <strong>階層型エージェント</strong> | 上位層のエージェントが作業を細分化し、下位層に委任。 | 「旅行キャンセル」依頼を、フライトキャンセル、ホテルキャンセル、レンタカーキャンセルに分割し、それぞれが別エージェントで対応。 |
| **マルチエージェントシステム(MAS)** | 複数の独立エージェントが協働（または競合）する。 | 協調：ホテル、フライト、娯楽の担当が分かれている。競合：複数エージェントがホテルの部屋を最良価格で提供するために競う。 |

## AIエージェントを使うべき時

使えるからといっていつも<em>使うべき</em>とは限りません。エージェントが特に有効な状況は以下の通りです：

![AIエージェントを使うべき時](../../../translated_images/ja/when-to-use-ai-agents.54becb3bed74a479.webp)

- <strong>オープンエンドな問題</strong> — 問題解決の手順が事前にプログラムできないとき。LLMに動的に道筋を考えさせる必要がある。
- <strong>複数ステップのプロセス</strong> — 一度の参照や生成ではなく、複数回にわたりツールを使う必要があるタスク。
- <strong>時間経過での改善</strong> — ユーザーフィードバックや環境のシグナルに基づき、システムを賢くしたいとき。

コースの後半「信頼できるAIエージェントの構築」レッスンで、AIエージェントの使うべき場合（と使わないべき場合）を詳しく掘り下げます。

## エージェント的ソリューションの基礎

### エージェント開発

エージェントを作るときに最初にやることは、<em>何ができるか</em>を定義することです。ツール、行動、振る舞いを決めます。

本コースでは、メインプラットフォームとして **Microsoft Foundry Agent Service** を使います。これにより以下が利用可能です：

- OpenAI、Mistral、Meta（Llama）などプロバイダーのモデル
- Tripadvisorなどのライセンスされたデータ
- 標準化されたOpenAPI 3.0ツール定義

### エージェント的パターン

LLMとはプロンプトを通じてやりとりします。エージェントでは全てのプロンプトを手作業で作成できるわけではありません。多段階で行動を起こすために、<strong>エージェント的パターン</strong>が使われます。これはLLMのプロンプトやオーケストレーションの再利用可能で拡張性のある戦略です。

本コースは最も一般的かつ有用なエージェント的パターンを軸に構成されています。

### エージェント的フレームワーク

エージェント的フレームワークは、開発者にテンプレート、ツール、インフラを提供し、エージェントの構築を容易にします。これにより：

- ツールや機能を接続しやすくし、
- エージェントの動作を監視（誤動作時のデバッグ）し、
- 複数のエージェント間での協働も可能にします。

本コースでは、製品レベルのエージェント構築に向けて **Microsoft Agent Framework (MAF)** に注力します。
