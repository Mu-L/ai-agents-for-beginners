# エージェンティックプロトコルの使用（MCP、A2A、NLWeb）

[![Agentic Protocols](../../../translated_images/ja/lesson-11-thumbnail.b6c742949cf1ce2a.webp)](https://youtu.be/X-Dh9R3Opn8)

> _(上の画像をクリックするとこのレッスンのビデオをご覧いただけます)_

AIエージェントの利用が増えるに従い、標準化・セキュリティの確保、そしてオープンイノベーションを支援するプロトコルの必要性も高まっています。本レッスンでは、このニーズに応える3つのプロトコル、Model Context Protocol (MCP)、Agent to Agent (A2A)、Natural Language Web (NLWeb)について解説します。

## はじめに

本レッスンで扱う内容は：

• <strong>MCP</strong>がAIエージェントに外部ツールとデータへのアクセスを許し、ユーザーのタスクを完了する方法。

• <strong>A2A</strong>が異なるAIエージェント間のコミュニケーションと協働を可能にする方法。

• <strong>NLWeb</strong>があらゆるウェブサイトに自然言語インターフェイスをもたらし、AIエージェントがコンテンツを発見し、対話できるようにする方法。

## 学習目標

• AIエージェントの文脈でMCP、A2A、NLWebの中核的な目的と利点を<strong>識別する</strong>こと。

• 各プロトコルがLLM、ツール、他のエージェント間の通信と相互作用をどのように促進するかを<strong>説明する</strong>こと。

• 複雑なエージェンティックシステム構築における各プロトコルの異なる役割を<strong>認識する</strong>こと。

## Model Context Protocol

<strong>Model Context Protocol (MCP)</strong>は、アプリケーションがLLMにコンテキストとツールを標準化して提供するためのオープン標準です。これにより、AIエージェントが異なるデータソースやツールに一貫した方法で接続できる「ユニバーサルアダプター」が実現されます。

MCPの構成要素、API直接利用と比較した利点、そしてAIエージェントがMCPサーバーを利用する例を見ていきましょう。

### MCPの主要コンポーネント

MCPは<strong>クライアント・サーバーアーキテクチャ</strong>で動作し、主要コンポーネントは以下です：

• <strong>ホスト</strong>はLLMアプリケーション（例：VSCodeのようなコードエディター）で、MCPサーバーへの接続を開始します。

• <strong>クライアント</strong>はホストアプリケーション内のコンポーネントで、サーバーとの一対一の接続を維持します。

• <strong>サーバー</strong>は特定の機能を公開する軽量プログラムです。

プロトコルには、MCPサーバーの機能である3つのコアプライミティブが含まれます：

• <strong>ツール</strong>：これはAIエージェントが呼び出して行動を実行できる個別のアクションや機能です。例えば、天気情報サービスが「天気を取得」ツールを公開したり、eコマースサーバーが「商品購入」ツールを公開したりします。MCPサーバーはそれぞれのツール名、説明、入出力スキーマを機能一覧に掲載します。

• <strong>リソース</strong>：MCPサーバーが提供できる読み取り専用のデータ項目やドキュメントであり、クライアントは必要に応じてそれらを取得できます。例としてファイル内容、データベースレコード、ログファイルなどがあります。リソースはテキスト（コードやJSONなど）かバイナリ（画像やPDFなど）です。

• <strong>プロンプト</strong>：これはあらかじめ定義されたテンプレートで、提案プロンプトを提供し、より複雑なワークフローを可能にします。

### MCPの利点

MCPはAIエージェントにとって次のような大きな利点を提供します：

• <strong>動的なツール発見</strong>：エージェントはサーバーから利用可能なツールのリストとその説明を動的に受け取ることができます。これは統合に静的コーディングが必要な従来APIとは異なり、API変更ごとにコードの更新が必要になる問題を解消します。MCPは「一度統合すればよい」アプローチを提供し、適応力を高めます。

• **LLM間の相互運用性**：MCPは異なるLLM間で動作し、よりよいパフォーマンスのためにコアモデルの切り替えが可能な柔軟性を提供します。

• <strong>標準化されたセキュリティ</strong>：MCPは標準的な認証方法を含み、追加のMCPサーバーアクセスを増やす際の拡張性を改善します。これは異なるキーや認証方式を管理する従来APIよりも簡単です。

### MCPの例

![MCP Diagram](../../../translated_images/ja/mcp-diagram.e4ca1cbd551444a1.webp)

たとえば、ユーザーがMCPで動くAIアシスタントを使ってフライト予約をしたい場合を想像してください。

1. <strong>接続</strong>：AIアシスタント（MCPクライアント）が航空会社のMCPサーバーに接続します。

2. <strong>ツール発見</strong>：クライアントは航空会社のMCPサーバーに「どんなツールがありますか？」と尋ねます。サーバーは「フライト検索」や「フライト予約」といったツールを返します。

3. <strong>ツール呼び出し</strong>：ユーザーがAIアシスタントに「ポートランドからホノルルへのフライトを検索して」と依頼すると、AIアシスタントはLLMを使い、「フライト検索」ツールを呼び出す必要があると判断し、出発地と目的地のパラメーターをMCPサーバーに渡します。

4. <strong>実行と応答</strong>：MCPサーバーはラッパーとして航空会社の内部予約APIを呼び出します。APIの応答（例：JSONデータ）を受け取ると、それをAIアシスタントに返送します。

5. <strong>さらなる対話</strong>：AIアシスタントはフライト候補を提示し、ユーザーがフライトを選択すると、同じMCPサーバーの「フライト予約」ツールを呼び出して予約を完了します。

## Agent-to-Agent Protocol (A2A)

MCPがLLMとツールをつなぐことに焦点を当てているのに対し、<strong>Agent-to-Agent (A2A) プロトコル</strong>は、異なるAIエージェント同士の通信と協力を可能にします。A2Aは異なる組織、環境、技術スタックのAIエージェントを接続し、共通のタスクを遂行します。

ここではA2Aの構成要素と利点、そして旅行アプリケーションでの適用例を見ていきましょう。

### A2Aの主要コンポーネント

A2Aはエージェント間の通信を可能にし、ユーザーのサブタスクを協力して完了させることに焦点を当てています。各コンポーネントはこれに寄与します：

#### エージェントカード

MCPサーバーがツール一覧を共有するのと似て、エージェントカードは：
- エージェント名。
- 完了する一般的なタスクの<strong>説明</strong>。
- 他のエージェント（または人間のユーザー）が、そのエージェントをいつ呼び出すべきかを理解する助けとなる<strong>具体的なスキル一覧</strong>と説明。
- エージェントの<strong>現在のエンドポイントURL</strong>。
- ストリーミング応答やプッシュ通知などの、エージェントの<strong>バージョン</strong>と<strong>機能</strong>。

#### エージェントエグゼキューター

エージェントエグゼキューターは<strong>ユーザーチャットのコンテキストをリモートエージェントに渡す役割</strong>を担います。リモートエージェントはタスクを理解するためにこれを必要とします。A2Aサーバーでは、エージェントは自身のLLMを使って受信したリクエストを解析し、自身の内部ツールでタスクを実行します。

#### アーティファクト

リモートエージェントがタスクを完了すると、その成果物がアーティファクトとして作成されます。アーティファクトは、<strong>エージェントの作業結果</strong>、<strong>完了内容の説明</strong>、そしてプロトコルを通じて送られる<strong>テキストコンテキスト</strong>を含みます。アーティファクト送信後は、再度必要になるまでリモートエージェントとの接続を終了します。

#### イベントキュー

このコンポーネントは<strong>更新の処理とメッセージの送受信</strong>に使用されます。特に生産環境のエージェンティックシステムでは、タスク完了までエージェント間の接続が切れないようにするために重要です。タスク完了までに時間がかかる場合がありますから。

### A2Aの利点

• <strong>強化された協働</strong>：異なるベンダーやプラットフォームのエージェントが相互にやり取りし、コンテキストを共有し、協力しあうことで、従来は分断されていたシステム間のシームレスな自動化を促進します。

• <strong>モデル選択の柔軟性</strong>：各A2Aエージェントはサービスに使うLLMを自由に選択でき、最適化やチューニングされたモデルを個別に用いることが可能です。これはMCPの一部シナリオでの単一LLM接続とは異なります。

• <strong>組み込み認証</strong>：認証はA2Aプロトコルに直接統合されており、エージェント間のやり取りに強固なセキュリティフレームワークを提供します。

### A2Aの例

![A2A Diagram](../../../translated_images/ja/A2A-Diagram.8666928d648acc26.webp)

旅行予約シナリオを拡張し、今回はA2Aを使ってみましょう。

1. <strong>ユーザーがマルチエージェントへリクエスト</strong>：「来週のホノルル旅行を全て予約してください。フライト、ホテル、レンタカー全て含めて」と「旅行代理人」A2Aクライアント／エージェントに話しかけます。

2. <strong>旅行代理人による調整</strong>：旅行代理人はこの複雑なリクエストを受け、自身のLLMでタスクを推論し、他の専門エージェントとやり取りすべきと判断します。

3. <strong>エージェント間通信</strong>：旅行代理人はA2Aプロトコルを使い、別会社が作った「航空会社エージェント」、「ホテルエージェント」、「レンタカーエージェント」など下流のエージェントに接続します。

4. <strong>タスクの委任実行</strong>：旅行代理人は「ホノルル行きのフライトを探して」「ホテルを予約して」「レンタカーを借りて」といった具体的タスクをそれぞれの専門エージェントに送信します。各専門エージェントは自前のLLMとツール（MCPサーバーもあり得ます）を用いて担当部分の予約を行います。

5. <strong>結果の統合応答</strong>：下流エージェント全てがタスクを完了すると、旅行代理人はフライト詳細、ホテル確認、レンタカー予約をまとめて、チャット形式の総合応答をユーザーに返します。

## Natural Language Web (NLWeb)

ウェブサイトはかねてからインターネット上の情報やデータアクセスの主要な手段です。

ここではNLWebの様々な構成要素、利点、そして旅行アプリケーションでの具体例を見ていきましょう。

### NLWebの構成要素

- **NLWebアプリケーション（コアサービスコード）**：自然言語の質問を処理するシステム。プラットフォームの異なる部分をつなげて応答を作り出します。ウェブサイトの自然言語機能を動かす<strong>エンジン</strong>と考えてください。

- **NLWebプロトコル**：ウェブサイトと自然言語でやりとりするための<strong>基本的なルールセット</strong>。JSON形式（通常はSchema.org活用）で応答を返します。HTMLがオンライン文書共有を可能にしたように、「AIウェブ」のシンプルな基盤構築を目指しています。

- **MCPサーバー（Model Context Protocolエンドポイント）**：各NLWeb構成は<strong>MCPサーバーとしても機能</strong>します。つまり、他のAIシステムとツール（「ask」メソッドなど）やデータを<strong>共有可能</strong>。実際には、ウェブサイトのコンテンツや機能をAIエージェントが利用できるようにし、サイトがより広い「エージェントエコシステム」の一部になります。

- <strong>埋め込みモデル</strong>：ウェブコンテンツを<strong>ベクトル（埋め込み）と呼ばれる数値表現に変換</strong>するモデル。これらのベクトルは意味を捉え、コンピュータが比較・検索できる形にします。専用データベースに保存し、ユーザーは使用する埋め込みモデルを選択可能です。

- **ベクトルデータベース（検索機構）**：ウェブサイトコンテンツの埋め込みを<strong>保存するデータベース</strong>。質問があれば、NLWebはこのベクトルデータベースを参照して最も関連性の高い情報を迅速に見つけます。類似度順の回答候補リストを返します。Qdrant、Snowflake、Milvus、Azure AI Search、Elasticsearchなど様々なベクトルストレージと連携可能です。

### NLWebの実例

![NLWeb](../../../translated_images/ja/nlweb-diagram.c1e2390b310e5fe4.webp)

先ほどの旅行予約ウェブサイトをもう一度考えますが、今回はNLWebで動作しています。

1. <strong>データ取り込み</strong>：旅行サイトの既存商品カタログ（例：フライトリスト、ホテル説明、ツアーパッケージ）がSchema.orgでフォーマットされるかRSSフィードとして読み込まれます。NLWebのツールがこの構造化データを取り込み、埋め込みを作成し、ローカルまたはリモートのベクトルデータベースに保存します。

2. **自然言語クエリ（人間ユーザー）**：ユーザーがサイトを訪れ、メニューを操作する代わりにチャットインターフェイスに「来週のホノルルで家族向けプール付きホテルを探して」と入力します。

3. **NLWebによる処理**：NLWebアプリがこのクエリを受け取ります。理解のためLLMに送信すると同時に、ベクトルデータベースを検索して関連ホテルを探します。

4. <strong>高精度結果</strong>：LLMはデータベースの検索結果を解釈し、「家族向け」「プールあり」「ホノルル」という条件に最適なマッチを特定し、自然言語で応答を整形します。重要な点は、応答は実際のホテルカタログに基づいており架空情報を避けていることです。

5. **AIエージェントとの相互作用**：NLWebはMCPサーバーとして機能するため、外部AI旅行エージェントもこのNLWebインスタンスに接続可能です。AIエージェントは`ask` MCPメソッドを使ってサイトに直接問い合わせ（例：`ask("ホノルル周辺でホテルが推薦するビーガン対応レストランはありますか？")`）ることができます。NLWebはこれを処理し、レストラン情報のデータベース（もし読み込まれていれば）を活用して構造化JSON応答を返します。
