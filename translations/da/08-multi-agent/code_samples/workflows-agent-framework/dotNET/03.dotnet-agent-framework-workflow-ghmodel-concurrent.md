# ⚡ Samtidige agentarbejdsgange med Azure OpenAI (Responses API) (.NET)

## 📋 Tutorial i højtydende parallel behandling

Dette notatblok demonstrerer **samtidige arbejdsgangsmønstre** ved brug af Microsoft Agent Framework til .NET og Azure OpenAI (Responses API). Du lærer, hvordan du bygger højtydende, parallelle arbejdsgange, der maksimerer gennemløb ved at køre flere AI-agenter samtidigt, mens der opretholdes koordinering og datakonsistens.

## 💻 Kørsel af koden

Den komplet implementering findes i `03.dotnet-agent-framework-workflow-ghmodel-concurrent.cs`. Denne fil demonstrerer en **Fan-Out/Fan-In samtidig arbejdsgang** til rejseplanlægning:

### 🏗️ **Arbejdsgangsarkitektur**

```
User Request → ConcurrentStartExecutor → [Researcher Agent || Planner Agent] → ConcurrentAggregationExecutor → Final Output
```

**Nøglekomponenter:**

1. **ConcurrentStartExecutor**: Sender brugerens anmodning til alle agenter samtidigt
2. **Researcher Agent**: Analyserer destinationer og seværdigheder samtidigt
3. **Planner Agent**: Opretter detaljerede rejseplaner samtidigt
4. **ConcurrentAggregationExecutor**: Samler og kombinerer resultater fra begge agenter

### 🎯 **Fan-Out/Fan-In mønster**

Denne arbejdsgang demonstrerer det klassiske **Fan-Out/Fan-In** mønster:
- **Fan-Out**: Én inputbesked sendes samtidig til flere agenter
- **Samtidig behandling**: Flere agenter arbejder parallelt på samme opgave
- **Fan-In**: Resultater fra alle agenter samles og aggregeres til et enkelt output

### 🔍 Forståelse af brugerdefinerede eksekveringsenheder

**ConcurrentStartExecutor:**
- Implementerer `IMessageHandler<string>` for at acceptere strenginput
- Sender beskeder til alle tilsluttede agenter
- Sender `TurnToken` for at udløse samtidig behandling

**ConcurrentAggregationExecutor:**
- Implementerer `IMessageHandler<ChatMessage>` for at modtage agentresponser
- Samler beskeder på trådsikker vis
- Agregerer, når alle forventede svar er modtaget
- Afleverer det endelige output ved brug af `context.YieldOutputAsync()`

### ⚡ Ydelsesfordele

**Samtidig vs Sekventiel:**
- Sekventiel: Agent1 (30s) → Agent2 (30s) = **60 sekunder i alt**
- Samtidig: Agent1 (30s) || Agent2 (30s) = **30 sekunder i alt**

**Gennemløbsforbedring**: Op til N× hurtigere for N samtidige agenter (afhængigt af arbejdsbyrde og ressourcer)

### 🛡️ Fejlhåndtering

Arbejdsgangen håndterer individuelle agentfejl yndefuldt:
- Hvis en agent fejler, fortsætter andre behandlingen
- Aggregatoren kan implementere timeout-logik
- Delvise resultater kan returneres efter behov
