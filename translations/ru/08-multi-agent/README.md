# Мультиагентные шаблоны проектирования

Как только вы начинаете работать над проектом, который включает несколько агентов, вам нужно рассмотреть мультиагентный шаблон проектирования. Однако может быть не сразу ясно, когда перейти к мультиагентам и каковы преимущества.

## Введение

В этом уроке мы постараемся ответить на следующие вопросы:

- В каких сценариях применимы мультиагенты?
- Каковы преимущества использования мультиагентов по сравнению с одним агентом, выполняющим несколько задач?
- Каковы строительные блоки для реализации мультиагентного шаблона проектирования?
- Как получить представление о взаимодействии нескольких агентов друг с другом?

## Цели обучения

После этого урока вы должны уметь:

- Определять сценарии, в которых применимы мультиагенты
- Осознавать преимущества использования мультиагентов по сравнению с одним агентом.
- Понимать строительные блоки для реализации мультиагентного шаблона проектирования.

Какова общая картина?

*Мультиагенты — это шаблон проектирования, который позволяет нескольким агентам работать вместе для достижения общей цели*.

Этот шаблон широко применяется в различных областях, включая робототехнику, автономные системы и распределённые вычисления.

## Сценарии применения мультиагентов

Так какие же сценарии являются хорошим примером использования мультиагентов? Ответ в том, что существует много сценариев, когда использование нескольких агентов полезно, особенно в следующих случаях:

- **Большие нагрузки**: Большие нагрузки можно разделить на более мелкие задачи и назначить их разным агентам, что позволяет выполнять обработку параллельно и быстрее завершать работу. Примером может служить задача по обработке больших объёмов данных.
- **Сложные задачи**: Сложные задачи, как и большие нагрузки, могут быть разбиты на более мелкие подзадачи и назначены разным агентам, каждый из которых специализируется на определённом аспекте задачи. Хороший пример — автономные транспортные средства, где разные агенты управляют навигацией, обнаружением препятствий и связью с другими автомобилями.
- **Разнообразие экспертиз**: Разные агенты могут обладать разной экспертизой, что позволяет им лучше справляться с разными аспектами задачи, чем один агент. Для этого случая хороший пример — здравоохранение, где агенты управляют диагностикой, планами лечения и мониторингом пациентов.

## Преимущества использования мультиагентов по сравнению с одним агентом

Система с одним агентом может хорошо работать для простых задач, но для более сложных задач использование нескольких агентов предоставляет несколько преимуществ:

- **Специализация**: Каждый агент может быть специализирован на конкретной задаче. Отсутствие специализации в одном агенте означает, что агент может делать всё, но может запутаться, что делать при сложной задаче. Например, он может выполнять задачу, для которой он не самый лучший.
- **Масштабируемость**: Системы легче масштабировать, добавляя больше агентов, чем перегружая одного агента.
- **Отказоустойчивость**: Если один агент выходит из строя, другие могут продолжать работу, обеспечивая надёжность системы.

Рассмотрим пример: забронировать поездку для пользователя. Система с одним агентом должна обрабатывать все аспекты бронирования — от поиска рейсов до бронирования отелей и автомобилей в аренду. Для этого одному агенту понадобятся инструменты для выполнения всех этих задач. Это может привести к сложной и монолитной системе, которую сложно поддерживать и масштабировать. Система с мультиагентами могла бы включать разных агентов, специализирующихся на поиске рейсов, бронировании отелей и аренде автомобилей. Это сделало бы систему более модульной, легче поддерживаемой и масштабируемой.

Сравните это с туристическим бюро в виде семейного бизнеса и турбюро во франшизе. Семейный бизнес имеет одного агента, который ведёт все аспекты бронирования, тогда как во франшизе разные агенты управляют разными аспектами бронирования.

## Строительные блоки реализации мультиагентного шаблона проектирования

Прежде чем реализовать мультиагентный шаблон проектирования, необходимо понять строительные блоки, из которых состоит этот шаблон.

Сделаем это более конкретным, снова рассмотрев пример бронирования поездки для пользователя. Строительные блоки в этом случае включают:

- **Взаимодействие агентов**: Агенты по поиску рейсов, бронированию отелей и автомобилей должны обмениваться информацией о предпочтениях и ограничениях пользователя. Нужно определить протоколы и методы для этой коммуникации. Конкретно это значит, что агент по поиску рейсов должен взаимодействовать с агентом по бронированию отелей, чтобы забронировать отель на те же даты, что и рейсы. Это значит, что агенты должны обмениваться информацией о датах поездки пользователя, нужно решить *какие агенты делятся информацией и как они это делают*.
- **Механизмы координации**: Агенты должны координировать свои действия, чтобы удовлетворять предпочтения и ограничения пользователя. Например, пользователь хочет, чтобы отель был близко к аэропорту, а при этом арендованные автомобили доступны только в аэропорту. Это значит, что агент по бронированию отелей должен координироваться с агентом по бронированию автомобилей, чтобы удовлетворять эти условия. Нужно определить *как агенты координируют свои действия*.
- **Архитектура агента**: Агенты должны иметь внутреннюю структуру для принятия решений и обучения на основе взаимодействия с пользователем. Это означает, что агент по поиску рейсов должен иметь структуру для принятия решений о том, какие рейсы рекомендовать. Нужно решить *как агенты принимают решения и учатся на взаимодействиях с пользователем*. Например, агент по поиску рейсов может использовать модель машинного обучения для рекомендаций, основанных на прошлых предпочтениях пользователя.
- **Видимость взаимодействия мультиагентной системы**: Нужно иметь видимость взаимодействий между агентами. Для этого нужны инструменты и методы для отслеживания активности и взаимодействий агентов: журналы, инструменты мониторинга, визуализации и метрики производительности.
- **Мультиагентные шаблоны**: Существуют различные шаблоны для реализации мультиагентных систем, например, централизованные, децентрализованные и гибридные архитектуры. Нужно выбрать подходящий шаблон для конкретного случая.
- **Человек в цикле**: В большинстве случаев в системе присутствует человек, и агентам нужно указать, когда запрашивать вмешательство человека. Например, пользователь может запросить конкретный отель или рейс, не рекомендованный агентами, или потребовать подтверждения бронирования.

## Видимость взаимодействия мультиагентной системы

Важно иметь видимость того, как множество агентов взаимодействуют друг с другом. Это необходимо для отладки, оптимизации и обеспечения эффективности системы. Для этого нужны инструменты и методы отслеживания активности и взаимодействия агентов: журналы, мониторинг, визуализации и метрики производительности.

Например, в случае бронирования поездки можно иметь панель управления, показывающую статус каждого агента, предпочтения и ограничения пользователя, а также взаимодействия между агентами. Панель может показывать даты поездки пользователя, рекомендуемые рейсы, отели и автомобили, что даст ясное представление о взаимодействии агентов и выполнении требований пользователя.

Рассмотрим каждый из этих аспектов более подробно.

- **Журналы и инструменты мониторинга**: Необходимо вести журнал каждого действия агента. Запись журнала может содержать информацию об агенте, совершившем действие, самом действии, времени и результате. Эта информация пригодится для отладки и оптимизации.

- **Инструменты визуализации**: Они помогают видеть взаимодействия между агентами интуитивно. Например, граф, показывающий поток информации между агентами, поможет обнаружить узкие места, неэффективность и другие проблемы.

- **Метрики производительности**: Они позволяют отслеживать эффективность мультиагентной системы. Например, время выполнения задачи, количество выполненных задач за единицу времени, точность рекомендаций агентов. Эти данные помогут определить области для улучшения и оптимизации.

## Мультиагентные шаблоны

Рассмотрим конкретные шаблоны, которые можно использовать для создания мультиагентных приложений. Вот несколько интересных вариантов:

### Групповой чат

Этот шаблон полезен, когда нужно создать приложение группового чата, в котором несколько агентов могут общаться друг с другом. Типичные случаи — командное сотрудничество, поддержка клиентов и социальные сети.

В этом шаблоне каждый агент представляет пользователя в групповом чате, а сообщения обмениваются между агентами с использованием протокола обмена сообщениями. Агенты могут отправлять сообщения в чат, получать их и отвечать другим агентам.

Этот шаблон можно реализовать с централизованной архитектурой, где все сообщения проходят через центральный сервер, или децентрализованной, где сообщения обмениваются напрямую.

![Групповой чат](../../../translated_images/ru/multi-agent-group-chat.ec10f4cde556babd.webp)

### Передача задачи

Этот шаблон полезен, когда требуется создать приложение, в котором несколько агентов передают друг другу задачи.

Типичные случаи — поддержка клиентов, управление задачами и автоматизация рабочих процессов.

В этом шаблоне каждый агент представляет задачу или шаг рабочего процесса, и агенты могут передавать задачи другим агентам на основе заданных правил.

![Передача задачи](../../../translated_images/ru/multi-agent-hand-off.4c5fb00ba6f8750a.webp)

### Коллаборативная фильтрация

Этот шаблон полезен, когда нужно создать приложение, где несколько агентов сотрудничают для предоставления рекомендаций пользователям.

Зачем несколько агентов сотрудничают? Потому что каждый может иметь разную экспертизу и вносить вклад в процесс рекомендации по-разному.

Рассмотрим пример, где пользователь хочет получить совет, какую акцию купить на фондовом рынке.

- **Эксперт отрасли**: Один агент может быть экспертом в конкретной отрасли.
- **Технический анализ**: Другой агент может быть экспертом в техническом анализе.
- **Фундаментальный анализ**: И ещё один агент может специализироваться на фундаментальном анализе. Сотрудничая, эти агенты смогут дать более всестороннюю рекомендацию пользователю.

![Рекомендация](../../../translated_images/ru/multi-agent-filtering.d959cb129dc9f608.webp)

## Сценарий: процесс возврата денег

Рассмотрим сценарий, когда клиент пытается получить возврат за товар, в этом процессе могут участвовать довольно много агентов. Разделим их на агентов, специфичных для этого процесса, и общих агентов, которые могут использоваться в других процессах.

**Агенты, специфичные для процесса возврата**:

Ниже перечислены агенты, которые могут участвовать в процессе возврата:

- **Агент клиента**: представляет клиента и отвечает за инициирование процесса возврата.
- **Агент продавца**: представляет продавца и отвечает за обработку возврата.
- **Агент платежей**: отвечает за возврат средств клиенту.
- **Агент разрешения проблем**: отвечает за разрешение возникающих в процессе возврата вопросов.
- **Агент соответствия**: обеспечивает соответствие процесса возврата нормативным требованиям и политикам.

**Общие агенты**:

Эти агенты могут использоваться в других частях вашего бизнеса.

- **Агент доставки**: отвечает за отправку товара обратно продавцу. Может использоваться как в процессе возврата, так и в общем процессе доставки товара по покупке.
- **Агент обратной связи**: собирает отзывы от клиентов, может работать в любое время, не только в процессе возврата.
- **Агент эскалации**: ответственен за передачу вопросов на более высокий уровень поддержки. Используется в любых процессах, где требуется эскалация.
- **Агент уведомлений**: отвечает за отправку уведомлений клиенту на разных этапах процесса возврата.
- **Агент аналитики**: анализирует данные, связанные с процессом возврата.
- **Агент аудита**: проверяет процесс возврата на соответствие и правильность исполнения.
- **Агент отчетности**: генерирует отчёты по процессу возврата.
- **Агент знаний**: поддерживает базу знаний по процессам возврата и другим аспектам бизнеса.
- **Агент безопасности**: обеспечивает безопасность процесса возврата.
- **Агент качества**: отвечает за контроль качества процесса возврата.

Выше перечислено немало агентов — и специфичных для возврата, и общих для других бизнес-процессов. Надеюсь, это даст вам представление, как выбрать агентов для вашей мультиагентной системы.

## Проверка знаний

### Вопрос 1

Какой сценарий лучше всего подходит для системы с множественными агентами?

- [ ] A1: Бот поддержки отвечает на часто задаваемые вопросы, используя одну базу знаний и небольшой набор инструментов.
- [ ] A2: Рабочий процесс возврата средств требует отдельные роли для борьбы с мошенничеством, платежей и соответствия, каждая со своими инструментами, и их результаты должны быть согласованы.
- [ ] A3: Один и тот же простой запрос на классификацию поступает тысячи раз в час.

### Вопрос 2

Когда обычно лучше выбрать одного агента?

- [ ] A1: Задачу можно выполнить с помощью одного набора инструкций и инструментов, без передачи специалистам.
- [ ] A2: Агент имеет доступ к более чем одному инструменту.
- [ ] A3: Рабочий процесс требует отдельных ролей с разными разрешениями и независимыми маршрутами аудита.
