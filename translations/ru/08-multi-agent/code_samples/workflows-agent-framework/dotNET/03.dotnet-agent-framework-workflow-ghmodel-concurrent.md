## 💻 Запуск кода

Полная реализация доступна в `03.dotnet-agent-framework-workflow-ghmodel-concurrent.cs`. Этот файл демонстрирует **параллельный рабочий процесс Fan-Out/Fan-In** для планирования путешествий:

### 🏗️ **Архитектура рабочего процесса**

```
User Request → ConcurrentStartExecutor → [Researcher Agent || Planner Agent] → ConcurrentAggregationExecutor → Final Output
```

**Ключевые компоненты:**

1. **ConcurrentStartExecutor**: Одновременная трансляция запроса пользователя всем агентам
2. **Researcher Agent**: Параллельный анализ направлений и достопримечательностей
3. **Planner Agent**: Параллельное создание детализированных планов путешествий
4. **ConcurrentAggregationExecutor**: Сбор и объединение результатов от обоих агентов

### 🎯 **Паттерн Fan-Out/Fan-In**

Этот рабочий процесс демонстрирует классический паттерн **Fan-Out/Fan-In**:
- **Fan-Out**: Один входной запрос рассылается одновременно нескольким агентам
- **Параллельная обработка**: Несколько агентов работают одновременно над одной задачей
- **Fan-In**: Результаты от всех агентов собираются и объединяются в единый вывод

### 📝 Ожидаемый результат

Рабочий процесс:
1. **Трансляция запроса**: Отправляет запрос "Спланировать поездку в Сиэтл в декабре" обоим агентам
2. **Параллельная обработка**: Оба агента работают одновременно:

4. **Отображение результатов**: Показ объединенного плана путешествия со всей информацией

### 🔍 Понимание пользовательских исполнителей (executors)

**ConcurrentStartExecutor:**
- Реализует `IMessageHandler<string>` для приема строкового ввода
- Транслирует сообщения всем подключенным агентам
- Отправляет `TurnToken` для запуска параллельной обработки

**ConcurrentAggregationExecutor:**
- Реализует `IMessageHandler<ChatMessage>` для приема ответов агентов
- Собирать сообщения в потокобезопасном режиме
- Производить агрегацию при получении всех ожидаемых ответов
- Возвращать финальный вывод через `context.YieldOutputAsync()`

### ⚡ Преимущества по производительности

**Параллельно vs Последовательно:**
- Последовательно: Agent1 (30с) → Agent2 (30с) = **60 секунд всего**
- Параллельно: Agent1 (30с) || Agent2 (30с) = **30 секунд всего**

**Увеличение пропускной способности**: До N раз быстрее при N параллельных агентах (в зависимости от нагрузки и ресурсов)

### 🛡️ Обработка ошибок

Рабочий процесс корректно обрабатывает сбои отдельных агентов:
- Если один агент выходит из строя, остальные продолжают работу
- Агрегатор может реализовывать логику тайм-аутов
- При необходимости могут возвращаться частичные результаты
