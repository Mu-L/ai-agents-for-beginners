## 💻 Запуск кода

Полная реализация доступна в файле `03.dotnet-agent-framework-workflow-ghmodel-concurrent.cs`. Этот файл демонстрирует **конкурентный вариант рабочего процесса Fan-Out/Fan-In** для планирования путешествий:

### 🏗️ **Архитектура рабочего процесса**

```
User Request → ConcurrentStartExecutor → [Researcher Agent || Planner Agent] → ConcurrentAggregationExecutor → Final Output
```

**Ключевые компоненты:**

1. **ConcurrentStartExecutor**: Рассылает запрос пользователя всем агентам одновременно
2. **Researcher Agent**: Анализирует направления и достопримечательности параллельно
3. **Planner Agent**: Создает детальные планы путешествий параллельно
4. **ConcurrentAggregationExecutor**: Собирает и объединяет результаты от обоих агентов

### 🎯 **Паттерн Fan-Out/Fan-In**

Этот рабочий процесс демонстрирует классический паттерн **Fan-Out/Fan-In**:
- **Fan-Out**: Один входной запрос рассылается нескольким агентам одновременно
- **Конкурентная обработка**: Несколько агентов параллельно работают над одной задачей
- **Fan-In**: Результаты всех агентов собираются и агрегируются в единый вывод

### 📝 Ожидаемый результат

Рабочий процесс выполнит:
1. **Трансляция запроса**: Отправит "Спланируй поездку в Сиэтл в декабре" обоим агентам
2. **Конкурентная обработка**: Оба агента работают одновременно:
   - Researcher выявляет достопримечательности и детали
   - Planner создает маршрут и организационные детали
3. **Агрегация**: Объединяет оба ответа в единый полный результат
4. **Отображение результатов**: Показывает объединённый план путешествия со всей информацией

### 🎯 Реальные применения

Этот конкурентный паттерн идеально подходит для:
- **Создания контента**: Несколько авторов создают разные части одновременно
- **Код-ревью**: Несколько рецензентов анализируют код с разных точек зрения
- **Маркетинговых исследований**: Параллельный анализ различных сегментов рынка
- **Обработки документов**: Одновременное извлечение, анализ и валидация
- **Многоперспективного анализа**: Получение разных точек зрения по одному входу

### 🔍 Понимание кастомных исполнителей

**ConcurrentStartExecutor:**
- Реализует `IMessageHandler<string>` для приёма строкового ввода
- Рассылает сообщения всем подключенным агентам
- Отправляет `TurnToken` для запуска конкурентной обработки

**ConcurrentAggregationExecutor:**
- Реализует `IMessageHandler<ChatMessage>` для получения ответов агентов
- Собирает сообщения потокобезопасно
- Выполняет агрегацию после получения всех ожидаемых ответов
- Отдаёт окончательный вывод через `context.YieldOutputAsync()`

### ⚡ Преимущества по производительности

**Конкурентно против последовательного:**
- Последовательно: Агент1 (30с) → Агент2 (30с) = **всего 60 секунд**
- Конкурентно: Агент1 (30с) || Агент2 (30с) = **всего 30 секунд**

**Увеличение пропускной способности**: До N× быстрее при N конкурентных агентах (в зависимости от нагрузки и ресурсов)

### 🛡️ Обработка ошибок

Рабочий процесс корректно обрабатывает сбои отдельных агентов:
- Если один агент падает, остальные продолжают обработку
- Агрегатор может реализовать логику таймаута
- При необходимости могут возвращаться частичные результаты
