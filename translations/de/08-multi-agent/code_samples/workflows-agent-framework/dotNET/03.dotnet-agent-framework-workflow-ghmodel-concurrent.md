## 💻 Ausführung des Codes

Die vollständige Implementierung finden Sie in `03.dotnet-agent-framework-workflow-ghmodel-concurrent.cs`. Diese Datei demonstriert einen **Fan-Out/Fan-In gleichzeitigen Workflow** für Reiseplanung:

### 🏗️ **Workflow-Architektur**

```
User Request → ConcurrentStartExecutor → [Researcher Agent || Planner Agent] → ConcurrentAggregationExecutor → Final Output
```

**Zentrale Komponenten:**

1. **ConcurrentStartExecutor**: Überträgt die Benutzeranfrage gleichzeitig an alle Agenten
2. **Researcher Agent**: Analysiert parallel Reiseziele und Attraktionen
3. **Planner Agent**: Erstellt parallel detaillierte Reisepläne
4. **ConcurrentAggregationExecutor**: Sammelt und vereint Ergebnisse beider Agenten

### 🎯 **Fan-Out/Fan-In Muster**

Dieser Workflow zeigt das klassische **Fan-Out/Fan-In** Muster:
- **Fan-Out**: Eine Eingabenachricht wird gleichzeitig an mehrere Agenten gesendet
- **Gleichzeitige Verarbeitung**: Mehrere Agenten arbeiten parallel an der gleichen Aufgabe
- **Fan-In**: Ergebnisse aller Agenten werden gesammelt und zu einer einzigen Ausgabe zusammengeführt

Der Workflow wird:
1. **Anfrage senden**: Sende "Plane eine Reise nach Seattle im Dezember" an beide Agenten
2. **Gleichzeitige Verarbeitung**: Beide Agenten arbeiten gleichzeitig:
   - Researcher identifiziert Attraktionen und Details

3. **Aggregation**: Kombiniert beide Antworten zu einer umfangreichen Ausgabe

### 🔍 Verständnis von benutzerdefinierten Executoren

**ConcurrentStartExecutor:**
- Implementiert `IMessageHandler<string>` zum Empfang von String-Eingaben
- Sendet Nachrichten an alle verbundenen Agenten
- Sendet `TurnToken`, um die gleichzeitige Verarbeitung zu starten

**ConcurrentAggregationExecutor:**
- Implementiert `IMessageHandler<ChatMessage>` zum Empfang von Agentenantworten
- Sammelt Nachrichten thread-sicher
- Aggregiert, wenn alle erwarteten Antworten eingetroffen sind
- Gibt die endgültige Ausgabe mit `context.YieldOutputAsync()` zurück

### ⚡ Leistungsverbesserungen

**Gleichzeitige vs sequentielle Ausführung:**
- Sequentiell: Agent1 (30s) → Agent2 (30s) = **60 Sekunden Gesamtzeit**
- Gleichzeitig: Agent1 (30s) || Agent2 (30s) = **30 Sekunden Gesamtzeit**

**Durchsatzsteigerung**: Bis zu N× schneller für N gleichzeitige Agenten (abhängig von Arbeitslast und Ressourcen)

### 🛡️ Fehlerbehandlung

Der Workflow handhabt einzelne Agentenfehler elegant:
- Wenn ein Agent ausfällt, fahren die anderen weiter mit der Verarbeitung
- Aggregator kann Timeout-Logik implementieren
- Teilresultate können bei Bedarf zurückgegeben werden
