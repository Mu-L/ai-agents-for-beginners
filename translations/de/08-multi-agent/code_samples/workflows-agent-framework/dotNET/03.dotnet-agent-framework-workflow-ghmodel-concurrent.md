# ⚡ Gleichzeitige Agenten-Workflows mit Azure OpenAI (Responses API) (.NET)

## 📋 Tutorial zur Hochleistungs-Parallelen Verarbeitung

## 💻 Ausführung des Codes

Die vollständige Implementierung finden Sie in `03.dotnet-agent-framework-workflow-ghmodel-concurrent.cs`. Diese Datei demonstriert einen **Fan-Out/Fan-In gleichzeitigen Workflow** für die Reiseplanung:

### 🏗️ **Workflow-Architektur**

```
User Request → ConcurrentStartExecutor → [Researcher Agent || Planner Agent] → ConcurrentAggregationExecutor → Final Output
```

**Wichtigste Komponenten:**

1. **ConcurrentStartExecutor**: Sendet die Benutzeranfrage gleichzeitig an alle Agenten
2. **Researcher Agent**: Analysiert gleichzeitig Reiseziele und Attraktionen
3. **Planner Agent**: Erarbeitet parallele detaillierte Reisepläne
4. **ConcurrentAggregationExecutor**: Sammelt und fasst Ergebnisse beider Agenten zusammen

### 🎯 **Fan-Out/Fan-In Muster**

Dieser Workflow zeigt das klassische **Fan-Out/Fan-In** Muster:
- **Fan-Out**: Eine Eingabenachricht wird gleichzeitig an mehrere Agenten gesendet
- **Gleichzeitige Verarbeitung**: Mehrere Agenten arbeiten parallel an derselben Aufgabe
- **Fan-In**: Ergebnisse aller Agenten werden gesammelt und zu einer einzigen Ausgabe aggregiert

Der Workflow wird:
1. **Anfrage senden**: „Plane eine Reise im Dezember nach Seattle“ an beide Agenten senden
2. **Gleichzeitige Verarbeitung**: Beide Agenten arbeiten parallel:

### 🔍 Verständnis von benutzerdefinierten Executors

**ConcurrentStartExecutor:**
- Implementiert `IMessageHandler<string>`, um String-Eingaben zu akzeptieren
- Sendet Nachrichten an alle verbundenen Agenten
- Sendet `TurnToken`, um die gleichzeitige Verarbeitung auszulösen

**ConcurrentAggregationExecutor:**
- Implementiert `IMessageHandler<ChatMessage>`, um Agentenantworten zu empfangen
- Sammelt Nachrichten thread-sicher
- Aggregiert sobald alle erwarteten Antworten eingetroffen sind
- Liefert finale Ausgabe mit `context.YieldOutputAsync()`

### ⚡ Leistungssteigerungen

**Parallel vs. sequenziell:**
- Sequenziell: Agent1 (30s) → Agent2 (30s) = **60 Sekunden gesamt**
- Parallel: Agent1 (30s) || Agent2 (30s) = **30 Sekunden gesamt**

**Durchsatzsteigerung**: Bis zu N× schneller für N gleichzeitige Agenten (abhängig von Arbeitslast und Ressourcen)

### 🛡️ Fehlerbehandlung

Der Workflow behandelt Einzelausfälle von Agenten elegant:
- Wenn ein Agent ausfällt, verarbeiten die anderen weiter
- Aggregator kann Timeout-Logik implementieren
- Ggf. können Teilergebnisse zurückgegeben werden
