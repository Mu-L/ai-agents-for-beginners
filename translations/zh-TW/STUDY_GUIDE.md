# 初學者的 AI 代理人 - 學習指南

使用此指南作為您進行課程時的實用夥伴。它並非用來取代課程內容，而是幫助您決定從哪裡開始、每堂課要注意什麼，以及如何將不同概念連結成一個小型的可運作代理人示範。




如果您是第一次接觸此處，請從簡單開始：

1. 閱讀 [課程設定](./00-course-setup/README.md)。
2. 按順序完成第01到06課。
3. 學習時心中保持一個小型示範想法。
4. 每完成一課，問自己：「我的代理人現在能做什麼，是之前做不到的？」


## 一個簡單的示範範例

學習代理人的好方法是在人示範想法中持續跟隨。

範例示範：<strong>一個課程助手代理人</strong>。

用戶問：

> 「我想了解代理人如何使用工具。找到合適的課程，總結我應該先讀什麼，並給我一個簡短的練習任務。」

3. <strong>規劃</strong>一條短期學習路徑，而非給出一長串答案。
4. <strong>利用對話上下文</strong>保持焦點在學習者的目標。

7. **在進行風險操作或使用敏感資料前，套用防護機制**。

學習每堂課時，回顧這個示範並問：這堂課會增加什麼新能力？




## 您的建構目標

課程結束時，您應能解釋並構建結合以下部分的代理人系統：


|------|------------------------|-------------|
| 模型 | 解讀使用者請求的推理引擎 | 理解學習者想要學習工具使用課程 |
| 工具 | 代理人可使用的函式、API、檔案、瀏覽器或服務 | 搜尋代碼庫或擷取課程內容 |
| 知識 | 用於回答依據的文件或資料 | 課程的 README 檔及教材 |
| 上下文 | 下次模型呼叫時包含的資訊 | 使用者目標與工具結果 |
| 記憶 | 儲存以備後續使用的資訊 | 學習者偏好動手做 Python 範例 |
| 規劃 | 將較大目標拆分成小步驟 | 找出課程，摘要內容，建議練習 |
| 編排 | 在工具、步驟或多代理間調度工作 | 規劃者呼叫搜尋工具，接著呼叫摘要器 |
| 信任 | 安全性、保護、評估與可觀察性 | 紀錄工具呼叫，且高影響操作前先詢問 |


## 模型與提供者

課程的程式範例使用 **Microsoft Agent Framework (MAF)** 並以 **Azure OpenAI Responses API** 為目標 — 這是建議使用的 API，結合聊天完成、工具調用、多模態輸入及有狀態對話於一個 API 表面。您可以經由 **Microsoft Foundry** 專案（使用 `FoundryChatClient`）或直接連接 Azure OpenAI（使用 `OpenAIChatClient`）。

學習課程時，您有幾個提供者選項：

- **Microsoft Foundry / Azure OpenAI (Responses API)** — 課程中使用的主要方式。使用`az login`登入以進行無密鑰的 Entra ID 驗證。
- **Foundry Local** — 透過 OpenAI 相容的 API，在本地端完整執行模型（無雲端、無 API 金鑰）。適合離線或無成本實驗。參見 [課程設定](./00-course-setup/README.md)。
- **MiniMax** — 提供具有大上下文模型的 OpenAI 相容服務器，可作為替代方案。

> **注意：** GitHub Models 已棄用（2026年7月退休），且不支援 Responses API。範例已更新為改用 Azure OpenAI / Microsoft Foundry。

## 初學者友善的核心理念

### 工具

工具是代理可呼叫以在模型外執行工作的項目。好的工具
擁有清晰名稱、狹義職責、型別化輸入、可預測的輸出，以及安全的
失敗機制。

以課程輔助示範來說，工具可能是：

- `search_lessons(query)`
- `read_lesson(path)`
- `create_practice_task(topic)`

### RAG 與知識

RAG 幫助代理從來源資料回答，而非猜測。在此
課程中，來源資料可能是課程 README、程式碼範例，
或從課程連結的外部資源。

當答案應基於文件、資料或當前
專案檔案時，請使用 RAG。

### 規劃

當請求有多個步驟時，規劃非常有用。保持計劃簡短且
讓開發者或使用者易於檢視。

對示範而言，計劃可能是：

1. 找出與工具使用相關的課程。
2. 摘要最相關的課程。
3. 推薦一個練習任務。

### 上下文

上下文是模型當下看到的資訊。上下文太少會讓代理
錯失重要細節。上下文太多會讓代理變慢、更昂貴，
或更容易混淆。

良好的上下文工程意指為下一次模型呼叫
選擇對的資訊。

### 記憶

記憶是為後續保留的訊息。不要保存所有東西。僅在有用、
安全且易於更新或刪除時保存資訊。

例如，記得「學習者偏好 Python 範例」可能有用。
記憶敏感個人資料通常則不建議。

### 評估與可觀察性

評估問：代理是否做了正確的事？

可觀察性問：我們能否看出過程？

對生產環境的代理，追蹤模型呼叫、工具呼叫、擷取的上下文、
延遲、成本、失敗與用戶反饋。

### 信任與安全

可信賴代理需要的不僅是有幫助的提示。使用最小權限工具、
高影響行為須有人類批准，必要時資料脫敏，及行為須稽核時的
日誌或收據。

## 15 分鐘複習流程

每課後使用此流程：

1. **用一句話總結本課內容。**
2. **命名新代理能力。** 例如：工具使用、檢索、
   規劃、記憶、可觀察性或安全性。
3. **加入課程輔助示範。** 現在示範有什麼變化？
4. **找出風險。** 若此能力被誤用可能出現哪些問題？
5. **寫一個測試題目。** 您如何檢驗代理行為良好？

## 快速自我檢查

繼續前請嘗試回答這些問題：

1. 代理可以做普通聊天機器人本身做不到的事是什麼？
2. 您的代理首先需要什麼工具？為什麼？
3. 哪個知識來源應作為代理回答的依據？
4. 下一次模型呼叫應包含哪些上下文？

5. 代理應該記住什麼，應該避免存儲什麼？
6. 代理什麼時候應該請求人工批准？
7. 什麼樣的日誌、追蹤或憑證能幫助你日後除錯或稽核代理？


## 建議的期末專題練習

在課程結束時，建立一個小型代理，幫助學習者瀏覽此
倉庫。

最低版本：

- 接受使用者輸入的主題。
- 找出最相關的課程。
- 摘要先閱讀的內容。
- 建議一個實作練習任務。
- 顯示使用了哪些課程檔案或連結。

進階版本：

- 記住學習者偏好的程式語言。
- 在回答前使用簡單計劃。
- 最終回應前加入自我檢查步驟。
- 記錄工具呼叫和檢索的來源。
- 在開啟瀏覽器或 UI 自動化任務前詢問確認。

這讓你有一個小但真實的方式，在一個專案中練習工具、RAG、規劃、
上下文、記憶、可觀察性和信任。
