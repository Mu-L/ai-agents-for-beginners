# ⚡ Lygiagretūs agentų darbo srautai su Azure OpenAI (Responses API) (.NET)

## 📋 Aukštos spartos lygiagrečios apdorojimo pamoka

Ši užrašų knygelė iliustruoja **lygiagrečius darbo srauto modelius** naudojant Microsoft Agent Framework .NET ir Azure OpenAI (Responses API). Išmoksite kurti aukštos spartos, lygiagrečius darbo srautus, kurie maksimaliai padidina pralaidumą vykdydami kelis DI agentus vienu metu, tuo pačiu palaikant koordinavimą ir duomenų vientisumą.

## 🎯 Mokymosi tikslai

### 🚀 **Lygiagretaus apdorojimo pagrindai**
- **Lygiagretus agentų vykdymas**: vienu metu paleiskite kelis DI agentus, kad pasiektumėte maksimalią spartos efektyvumą
- **Async/Await modeliai**: naudokite .NET asinchroninio programavimo modelį efektyviai lygiagrečiai veiklai
- **Azure OpenAI (Responses API)**: koordinuokite daugybę vienu metu vykdomų užklausų Azure OpenAI Responses API
- **Išteklių valdymas**: efektyviai valdykite DI modelių išteklius lygiagrečiose operacijose

### 🏗️ **Pažangi lygiagrečios veiklos architektūra**
- **Užduočių pagrindu veikianti paralelizacija**: naudokite .NET Task Parallel Library optimaliam lygiagretumui
- **Sinchronizavimo modeliai**: koordinuokite lygiagrečius agentus, išvengdami konkurencinių sąlygų
- **Darbo paskirstymas**: efektyviai paskirstykite darbą turimų lygiagrečios apdorojimo pajėgumų atžvilgiu
- **Atsparumas klaidoms**: tvarkykite atskirų agentų gedimus nekliudant visam darbo srautui

### 🏢 **Įmonių lygiagretios programos**
- **Didelio tūrio dokumentų apdorojimas**: vienu metu apdorokite kelis dokumentus
- **Realaus laiko turinio analizė**: vienu metu analizuokite gaunamus duomenų srautus
- **Partijų apdorojimo optimizavimas**: maksimaliai padidinkite pralaidumą didelio masto duomenų apdorojimui
- **Daugiakanalė analizė**: lygiagretus skirtingų turinio tipų ir formatų apdorojimas

## 💻 Kodo paleidimas

Pilnas įgyvendinimas pateiktas faile `03.dotnet-agent-framework-workflow-ghmodel-concurrent.cs`. Šis failas demonstruoja **Fan-Out/Fan-In lygiagrečių darbo srautą** kelionių planavimui:

### 🏗️ **Darbo srauto architektūra**

```
User Request → ConcurrentStartExecutor → [Researcher Agent || Planner Agent] → ConcurrentAggregationExecutor → Final Output
```

**Pagrindinės dalys:**

1. **ConcurrentStartExecutor**: vienu metu transliuoja vartotojo užklausą visiems agentams
2. **Researcher Agent**: lygiagrečiai analizuoja kelionės tikslus ir lankytinas vietas
3. **Planner Agent**: kuria detalius kelionės planus lygiagrečiai
4. **ConcurrentAggregationExecutor**: surenka ir sujungia abiejų agentų rezultatus

### 🎯 **Fan-Out/Fan-In modelis**

Šis darbo srautas demonstruoja klasikinius **Fan-Out/Fan-In** modelio principus:
- **Fan-Out**: vienas įvesties pranešimas transliuojamas vienu metu keliems agentams
- **Lygiagretus apdorojimas**: keli agentai vienu metu atlieka tą pačią užduotį
- **Fan-In**: iš visų agentų surinkti rezultatai apjungiami į vieną išvestį

### 🎯 Realūs taikymai

Šis lygiagretaus darbo srauto modelis pritaikomas:
- **Turinio kūrimas**: keli rašytojai kuriantys skirtingas dalis vienu metu
- **Kodo peržiūra**: keli peržiūrėtojai analizuojantys kodą iš skirtingų perspektyvų
- **Rinkos tyrimai**: lygiagreti skirtingų rinkos segmentų analizė
- **Dokumentų apdorojimas**: lygiagretus ištraukimų, analizės ir patikrinimo vykdymas
- **Daugiaperspektyvinė analizė**: skirtingi požiūriai į tą pačią įvestį

### 🔍 Suprasti individualius vykdytojus (Executor)

**ConcurrentStartExecutor:**
- Įgyvendina `IMessageHandler<string>`, priimdamas tekstinę įvestį
- Transliuoja pranešimus visiems prijungtiems agentams
- Išsiunčia `TurnToken`, inicijuodamas lygiagretų apdorojimą

**ConcurrentAggregationExecutor:**
- Įgyvendina `IMessageHandler<ChatMessage>`, gaudamas agentų atsakymus
- Surenka pranešimus saugiai daugiagyslėje aplinkoje
- Vykdo suvestinę tada, kai visos numatomos atsakymų dalys gautos
- Sugeneruoja galutinę išvestį naudodamas `context.YieldOutputAsync()`

### ⚡ Našumo privalumai

**Lygiagretus vs sekvencinis:**
- Sekvencinis: Agent1 (30s) → Agent2 (30s) = **iš viso 60 sekundžių**
- Lygiagretus: Agent1 (30s) || Agent2 (30s) = **iš viso 30 sekundžių**

**Pralaidumo pagerėjimas**: N kartų greitesnis, jei yra N lygiagrečių agentų (priklauso nuo apkrovos ir išteklių)

### 🛡️ Klaidų valdymas

Darbo srautas sklandžiai tvarko atskirų agentų gedimus:
- jei vienas agentas sugedo, kiti toliau dirba
- suvestinė gali įgyvendinti laiko limitų logiką
- prireikus gali būti grąžinami daliniai rezultatai
