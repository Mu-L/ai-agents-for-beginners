## 💻 Exécution du code

L’implémentation complète est disponible dans `03.dotnet-agent-framework-workflow-ghmodel-concurrent.cs`. Ce fichier démontre un **flux de travail concurrent Fan-Out/Fan-In** pour la planification de voyages :

### 🏗️ **Architecture du flux de travail**

```
User Request → ConcurrentStartExecutor → [Researcher Agent || Planner Agent] → ConcurrentAggregationExecutor → Final Output
```

**Composants clés :**

1. **ConcurrentStartExecutor** : Diffuse simultanément la requête utilisateur à tous les agents
2. **Agent Researcher** : Analyse les destinations et attractions en parallèle
3. **Agent Planner** : Crée des plans de voyage détaillés simultanément
4. **ConcurrentAggregationExecutor** : Collecte et fusionne les résultats des deux agents

### 🎯 **Patron Fan-Out/Fan-In**

Ce flux de travail illustre le patron classique **Fan-Out/Fan-In** :
- **Fan-Out** : Un message d’entrée est diffusé simultanément à plusieurs agents
- **Traitement concurrent** : Plusieurs agents travaillent en parallèle sur la même tâche
- **Fan-In** : Les résultats de tous les agents sont collectés et agrégés en une sortie unique

### 📝 Résultat attendu

Le flux de travail va :
1. **Diffuser la requête** : Envoyer "Planifie un voyage à Seattle en décembre" aux deux agents
2. **Traitement concurrent** : Les deux agents travaillent simultanément :

### 🔍 Comprendre les exécutants personnalisés

**ConcurrentStartExecutor :**
- Implémente `IMessageHandler<string>` pour accepter une entrée string
- Diffuse les messages à tous les agents connectés
- Envoie un `TurnToken` pour déclencher le traitement concurrent

**ConcurrentAggregationExecutor :**
- Implémente `IMessageHandler<ChatMessage>` pour recevoir les réponses des agents
- Collecte les messages de manière thread-safe
- Agrège lorsque toutes les réponses attendues arrivent
- Produit la sortie finale avec `context.YieldOutputAsync()`

### ⚡ Avantages en performance

**Concurrent vs Séquentiel :**
- Séquentiel : Agent1 (30s) → Agent2 (30s) = **60 secondes au total**
- Concurrent : Agent1 (30s) || Agent2 (30s) = **30 secondes au total**

**Amélioration du débit** : Jusqu’à N× plus rapide pour N agents concurrents (en fonction de la charge et des ressources)

### 🛡️ Gestion des erreurs

Le flux de travail gère gracieusement les pannes des agents individuels :
- Si un agent échoue, les autres continuent le traitement
- L’agrégateur peut implémenter une logique de temporisation
- Des résultats partiels peuvent être retournés si nécessaire

### 📊 Fonctionnalités avancées

**Nombre dynamique d’agents :**
Modifiez la logique d’agrégation pour supporter un nombre variable d’agents :

```csharp
private int _expectedAgentCount;
private readonly List<ChatMessage> _messages = [];

public async ValueTask HandleAsync(ChatMessage message, IWorkflowContext context)
{
    this._messages.Add(message);
    if (this._messages.Count == _expectedAgentCount)
    {
        // Process aggregation
    }
}
```

Ce patron de flux de travail concurrent est essentiel pour construire des systèmes d’agents IA performants et évolutifs !
