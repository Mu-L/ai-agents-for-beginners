# ⚡ เวิร์กโฟลว์ Agent พร้อมกันด้วย Azure OpenAI (Responses API) (.NET)

## 📋 บทช่วยสอนการประมวลผลขนานที่มีประสิทธิภาพสูง

โน้ตบุ๊กนี้สาธิต **รูปแบบเวิร์กโฟลว์พร้อมกัน** โดยใช้ Microsoft Agent Framework สำหรับ .NET และ Azure OpenAI (Responses API) คุณจะได้เรียนรู้วิธีสร้างเวิร์กโฟลว์ประมวลผลแบบขนานที่มีประสิทธิภาพสูง ซึ่งเพิ่มปริมาณงานสูงสุดโดยการรันเอเย่นต์ AI หลายตัวพร้อมกันในขณะที่ยังคงรักษาการประสานงานและความสอดคล้องของข้อมูลไว้

## 🎯 วัตถุประสงค์การเรียนรู้

### 🚀 **พื้นฐานการประมวลผลพร้อมกัน**
- **การรันเอเย่นต์แบบขนาน**: รันเอเย่นต์ AI หลายตัวพร้อมกันเพื่อประสิทธิภาพสูงสุด
- **รูปแบบ Async/Await**: ใช้โมเดลการเขียนโปรแกรมแบบ async ของ .NET เพื่อประสิทธิภาพในการทำงานพร้อมกัน
- **Azure OpenAI (Responses API)**: ประสานงานการเรียก Azure OpenAI Responses API หลายครั้งพร้อมกัน
- **การจัดการทรัพยากร**: จัดการทรัพยากรโมเดล AI อย่างมีประสิทธิภาพในระหว่างการทำงานพร้อมกัน

### 🏗️ **สถาปัตยกรรมความพร้อมกันขั้นสูง**
- **งานแบบขนานโดยใช้ Task**: ใช้ .NET Task Parallel Library สำหรับการรันพร้อมกันที่เหมาะสมที่สุด
- **รูปแบบการประสานงาน**: ประสานงานเอเย่นต์พร้อมกันในขณะที่หลีกเลี่ยงปัญหา race condition
- **การกระจายโหลด**: แจกจ่ายงานอย่างมีประสิทธิภาพตามความสามารถการประมวลผลพร้อมกันที่มี
- **ความทนทานต่อความผิดพลาด**: จัดการความล้มเหลวของเอเย่นต์แต่ละตัวโดยไม่หยุดเวิร์กโฟลว์ทั้งหมด

### 🏢 **แอปพลิเคชันพร้อมกันระดับองค์กร**
- **การประมวลผลเอกสารจำนวนมาก**: ประมวลผลงานเอกสารหลายฉบับพร้อมกัน
- **การวิเคราะห์เนื้อหาแบบเรียลไทม์**: วิเคราะห์สตรีมข้อมูลที่เข้ามาหลายแหล่งพร้อมกัน
- **การเพิ่มประสิทธิภาพการประมวลผลแบบกลุ่ม**: เพิ่มปริมาณงานสูงสุดสำหรับการประมวลผลข้อมูลขนาดใหญ่
- **การวิเคราะห์หลายโหมด**: ประมวลผลพร้อมกันกับประเภทเนื้อหาและฟอร์แมตที่ต่างกัน

## 💻 การรันโค้ด

การใช้งานครบถ้วนมีในไฟล์ `03.dotnet-agent-framework-workflow-ghmodel-concurrent.cs` ไฟล์นี้สาธิตเวิร์กโฟลว์แบบ **Fan-Out/Fan-In พร้อมกัน** สำหรับการวางแผนการเดินทาง:

### 🏗️ **สถาปัตยกรรมเวิร์กโฟลว์**

```
User Request → ConcurrentStartExecutor → [Researcher Agent || Planner Agent] → ConcurrentAggregationExecutor → Final Output
```

**ส่วนประกอบหลัก:**

1. **ConcurrentStartExecutor**: กระจายคำขอของผู้ใช้ไปยังเอเย่นต์ทั้งหมดพร้อมกัน
2. **Researcher Agent**: วิเคราะห์จุดหมายและสถานที่ท่องเที่ยวพร้อมกัน
3. **Planner Agent**: สร้างแผนการเดินทางอย่างละเอียดพร้อมกัน
4. **ConcurrentAggregationExecutor**: เก็บรวบรวมและรวมผลลัพธ์จากเอเย่นต์ทั้งสอง

### 🎯 **รูปแบบ Fan-Out/Fan-In**

เวิร์กโฟลว์นี้แสดงรูปแบบ **Fan-Out/Fan-In** แบบคลาสสิก:
- **Fan-Out**: ข้อความอินพุตหนึ่งข้อความถูกกระจายไปยังเอเย่นต์หลายตัวพร้อมกัน
- **การประมวลผลพร้อมกัน**: เอเย่นต์หลายตัวทำงานพร้อมกันในงานเดียวกัน
- **Fan-In**: รวบรวมและรวมผลลัพธ์จากเอเย่นต์ทั้งหมดเข้าเป็นผลลัพธ์เดียว

### 📝 ผลลัพธ์ที่คาดหวัง

เวิร์กโฟลว์จะ:
1. **ส่งคำขอแบบแพร่ภาพ**: ส่ง "วางแผนการเดินทางไปซีแอตเทิลในเดือนธันวาคม" ไปยังเอเย่นต์ทั้งสอง
2. **การประมวลผลพร้อมกัน**: เอเย่นต์ทั้งสองทำงานพร้อมกัน:
   - นักวิจัยระบุสถานที่ท่องเที่ยวและรายละเอียด
   - นักวางแผนสร้างกำหนดการเดินทางและโลจิสติกส์
3. **การรวบรวมผลลัพธ์**: รวมคำตอบทั้งสองเป็นผลลัพธ์ที่ครอบคลุม
4. **แสดงผลลัพธ์**: แสดงแผนการเดินทางที่รวมผลลัพธ์ทั้งหมด

    .Build();

### 🎯 การใช้งานในโลกจริง

รูปแบบการพร้อมกันนี้เหมาะสำหรับ:
- **การสร้างเนื้อหา**: ผู้เขียนหลายคนสร้างส่วนต่างๆ พร้อมกัน
- **การตรวจสอบโค้ด**: ผู้ตรวจหลายคนวิเคราะห์โค้ดจากมุมมองที่แตกต่างกัน
- **การวิจัยตลาด**: การวิเคราะห์แบบขนานของกลุ่มตลาดต่างๆ
- **การประมวลผลเอกสาร**: การสกัด วิเคราะห์ และตรวจสอบพร้อมกัน
- **การวิเคราะห์มุมมองหลายด้าน**: รับมุมมองต่างๆ ในอินพุตเดียวกัน

### 🔍 ทำความเข้าใจ Custom Executors

**ConcurrentStartExecutor:**
- ใช้ `IMessageHandler<string>` เพื่อรับข้อมูลประเภทสตริง
- กระจายข้อความไปยังเอเย่นต์ที่เชื่อมต่อทั้งหมด
- ส่ง `TurnToken` เพื่อกระตุ้นการประมวลผลพร้อมกัน

**ConcurrentAggregationExecutor:**
- ใช้ `IMessageHandler<ChatMessage>` เพื่อรับคำตอบจากเอเย่นต์
- เก็บรวบรวมข้อความแบบปลอดภัยต่อเธรด
- รวมผลลัพธ์เมื่อได้รับคำตอบครบถ้วน
- ส่งผลลัพธ์สุดท้ายโดยใช้ `context.YieldOutputAsync()`

### ⚡ ประโยชน์ด้านประสิทธิภาพ

**พร้อมกัน vs ลำดับ:**
- ลำดับ: Agent1 (30 วินาที) → Agent2 (30 วินาที) = **60 วินาทีรวม**
- พร้อมกัน: Agent1 (30 วินาที) || Agent2 (30 วินาที) = **30 วินาทีรวม**

**การปรับปรุงอัตราผ่าน**: เร็วขึ้นสูงสุด N× สำหรับเอเย่นต์พร้อมกัน N ตัว (ขึ้นกับปริมาณงานและทรัพยากร)

### 🛡️ การจัดการข้อผิดพลาด
