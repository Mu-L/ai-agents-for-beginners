# ⚡ เวิร์กโฟลว์เอเจนต์แบบขนานด้วย Azure OpenAI (Responses API) (.NET)

## 📋 แบบฝึกหัดการประมวลผลคู่ขนานประสิทธิภาพสูง

โน้ตบุ๊กนี้แสดงตัวอย่าง **รูปแบบเวิร์กโฟลว์แบบขนาน** โดยใช้ Microsoft Agent Framework สำหรับ .NET และ Azure OpenAI (Responses API) คุณจะได้เรียนรู้วิธีสร้างเวิร์กโฟลว์ที่ประมวลผลคู่ขนานที่มีประสิทธิภาพสูงซึ่งเพิ่มประสิทธิภาพการส่งผ่านข้อมูลสูงสุดด้วยการเรียกใช้งานเอเจนต์ปัญญาประดิษฐ์หลายตัวพร้อมกันในขณะที่ยังคงความประสานงานและความสอดคล้องของข้อมูลไว้

## 🎯 วัตถุประสงค์การเรียนรู้

### 🚀 **พื้นฐานการประมวลผลคู่ขนาน**
- **การเรียกใช้งานเอเจนต์แบบขนาน**: เรียกใช้งานเอเจนต์ AI หลายตัวพร้อมกันเพื่อประสิทธิภาพสูงสุด
- **รูปแบบ Async/Await**: ใช้โมเดลการเขียนโปรแกรม async ของ .NET เพื่อความมีประสิทธิภาพในการทำงานคู่ขนาน
- **Azure OpenAI (Responses API)**: ประสานงานการเรียกใช้งาน Azure OpenAI Responses API หลายครั้งพร้อมกัน
- **การจัดการทรัพยากร**: จัดการทรัพยากรโมเดล AI อย่างมีประสิทธิภาพในระหว่างการทำงานคู่ขนาน

### 🏗️ **สถาปัตยกรรมการทำงานคู่ขนานขั้นสูง**
- **การทำงานคู่ขนานแบบใช้ Task**: ใช้ .NET Task Parallel Library สำหรับการทำงานคู่ขนานที่เหมาะสมที่สุด
- **รูปแบบการประสานงาน**: ประสานงานเอเจนต์คู่ขนานโดยหลีกเลี่ยงสภาวะแข่งขัน
- **การกระจายโหลด**: แจกจ่ายงานอย่างมีประสิทธิภาพในความจุการประมวลผลคู่ขนานที่มีอยู่
- **ความทนทานต่อข้อผิดพลาด**: จัดการข้อผิดพลาดของเอเจนต์แต่ละตัวโดยไม่หยุดเวิร์กโฟลว์ทั้งระบบ

### 🏢 **แอปพลิเคชันคู่ขนานระดับองค์กร**
- **การประมวลผลเอกสารปริมาณมาก**: ประมวลผลงานหลายเอกสารพร้อมกัน
- **การวิเคราะห์เนื้อหาแบบเรียลไทม์**: วิเคราะห์ข้อมูลในสตรีมขาเข้าหลายรายการแบบคู่ขนาน
- **การปรับแต่งการประมวลผลแบบแบตช์**: เพิ่มประสิทธิภาพการผ่านข้อมูลสำหรับการประมวลผลข้อมูลขนาดใหญ่
- **การวิเคราะห์มัลติโมดัล**: การประมวลผลคู่ขนานของเนื้อหาประเภทและรูปแบบต่างๆ

- **การกระจายโหลด**: การแจกจ่ายงานอย่างเท่าเทียมในกลุ่มเอเจนต์ที่มีอยู่

## 💻 การรันโค้ด

การใช้งานโค้ดเต็มรูปแบบอยู่ในไฟล์ `03.dotnet-agent-framework-workflow-ghmodel-concurrent.cs` ไฟล์นี้แสดงตัวอย่าง **เวิร์กโฟลว์คู่ขนานแบบ Fan-Out/Fan-In** สำหรับการวางแผนการเดินทาง:

### 🏗️ **สถาปัตยกรรมเวิร์กโฟลว์**

```
User Request → ConcurrentStartExecutor → [Researcher Agent || Planner Agent] → ConcurrentAggregationExecutor → Final Output
```

**ส่วนประกอบหลัก:**

1. **ConcurrentStartExecutor**: ส่งคำขอของผู้ใช้ไปยังเอเจนต์ทั้งหมดพร้อมกัน
2. **Researcher Agent**: วิเคราะห์จุดหมายและสถานที่ท่องเที่ยวแบบคู่ขนาน
3. **Planner Agent**: สร้างแผนการเดินทางโดยละเอียดแบบคู่ขนาน
4. **ConcurrentAggregationExecutor**: รวบรวมและรวมผลลัพธ์จากทั้งสองเอเจนต์

### 🎯 **รูปแบบ Fan-Out/Fan-In**

เวิร์กโฟลว์นี้แสดงรูปแบบคลาสสิกของ **Fan-Out/Fan-In**:
- **Fan-Out**: ข้อความอินพุตหนึ่งรายการถูกแพร่กระจายไปยังเอเจนต์หลายตัวพร้อมกัน
- **การประมวลผลคู่ขนาน**: เอเจนต์หลายตัวทำงานคู่ขนานกับงานเดียวกัน
- **Fan-In**: รวบรวมผลจากเอเจนต์ทุกตัวและรวมเป็นผลลัพธ์เดียว

### 📝 ผลลัพธ์ที่คาดหวัง

เวิร์กโฟลว์จะ:
1. **แพร่กระจายคำขอ**: ส่งข้อความ "วางแผนทริปไปซีแอตเทิลในเดือนธันวาคม" ไปยังเอเจนต์ทั้งสอง
2. **ประมวลผลคู่ขนาน**: ทั้งสองเอเจนต์ทำงานพร้อมกัน:
   - Researcher ระบุสถานที่ท่องเที่ยวและรายละเอียด
   - Planner สร้างแผนการเดินทางและโลจิสติกส์
3. **การรวมผลลัพธ์**: รวมคำตอบทั้งสองเป็นผลลัพธ์ที่ครบถ้วน
4. **แสดงผล**: แสดงแผนการเดินทางแบบรวมทั้งหมด

- **การประมวลผลเอกสาร**: การดึงข้อมูล วิเคราะห์ และตรวจสอบแบบคู่ขนาน
- **การวิเคราะห์หลากหลายมุมมอง**: รับมุมมองที่หลากหลายจากอินพุตเดียวกัน

### 🔍 ความเข้าใจ Custom Executors

**ConcurrentStartExecutor:**
- ใช้ `IMessageHandler<string>` เพื่อรับอินพุตเป็นสตริง
- ส่งข้อความไปยังเอเจนต์ที่เชื่อมต่อทั้งหมด
- ส่ง `TurnToken` เพื่อเริ่มประมวลผลคู่ขนาน

**ConcurrentAggregationExecutor:**
- ใช้ `IMessageHandler<ChatMessage>` เพื่อรับคำตอบจากเอเจนต์
- รวบรวมข้อความอย่างปลอดภัยในหลายเธรด
- รวมผลลัพธ์เมื่อได้รับคำตอบครบถ้วน
- ส่งผลลัพธ์สุดท้ายโดยใช้ `context.YieldOutputAsync()`

### ⚡ ประโยชน์ด้านประสิทธิภาพ

**แบบคู่ขนานเทียบกับแบบเรียงลำดับ:**
- เรียงลำดับ: Agent1 (30วินาที) → Agent2 (30วินาที) = **รวม 60 วินาที**
- คู่ขนาน: Agent1 (30วินาที) || Agent2 (30วินาที) = **รวม 30 วินาที**

**ปรับปรุงการส่งผ่านข้อมูล**: เร็วขึ้นสูงสุด N เท่าสำหรับเอเจนต์คู่ขนาน N ตัว (ขึ้นอยู่กับภาระงานและทรัพยากร)
