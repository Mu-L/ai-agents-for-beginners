# Тестування агентів «на дим»

Ця папка містить **каталоги smoke-тестів** для агентів, яких ви створюєте на курсі.
Smoke-тест — це дешевий, швидкий перевірочний тест, що **розгорнутий агент Microsoft Foundry, розміщений
у хмарі**, є доступним, відповідає і виконує найпростіші очікування від запиту.
Це перший пропускний контроль — а не заміна повного процесу оцінювання,
який ви вивчаєте у [Уроці 10](../10-ai-agents-production/README.md) та
[Уроці 16](../16-deploying-scalable-agents/README.md).

Каталоги використовуються дією GitHub [AI Smoke Test](https://github.com/marketplace/actions/ai-smoke-test)
через робочий процес [`.github/workflows/smoke-test.yml`](../../../.github/workflows/smoke-test.yml).


## Як запустити

1. **Розгорніть агента уроку** в Microsoft Foundry як розміщеного агента (див.
   Урок 16 для робочого процесу розгортання). Запам'ятайте **назву агента** та
   **endpoint проекту Foundry**.
2. Додайте ці секрети репозиторію (Налаштування → Секрети та змінні → Дії):
   `AZURE_CLIENT_ID`, `AZURE_TENANT_ID`, `AZURE_SUBSCRIPTION_ID`. Федеративна
   ідентичність потребує ролі **Azure AI User** на рівні **області проекту Foundry**.
3. У вкладці **Actions** запустіть **Smoke-test hosted agents** і оберіть
   `tests_file` для уроку, після чого введіть відповідний `agent_name` та
   `project_endpoint`.

## Для яких уроків є smoke-тести?

Smoke-тести застосовуються до уроків, де ви **розгортаєте агента**, відповіді якого у вигляді тексту можна
перевірити на відповідність відомому вмісту. Уроки, які є концептуальними, працюють лише локально,
або генерують недетерміновані креативні виходи, навмисно виключені:

- **Урок 17 (Створення локальних AI агентів)** виконується цілком на вашій робочій станції з
  Foundry Local і **не** надає endpoint для Foundry Responses, тому ця
  дія не застосовується. Перевірте його, запустивши ноутбук локально.
- Уроки з патернами проєктування та теорії (02, 03, 06, 07, 09, 12) не постачають жодного
  агента для розгортання і smoke-тестування.

## Схема каталогу (швидкий довідник)

Кожен каталог є JSON документом з масивом верхнього рівня `tests`. Кожен запис виконує POST
одного запиту і робить перевірку відповіді:

| Поле | Значення |
|-------|---------|
| `id` | Унікальний ідентифікатор кроку, виводиться у журналі. |
| `description` | Людинозрозумілий опис призначення. |
| `prompt` | Повідомлення, що відправляється агенту. |
| `assertions.status` | Очікуваний HTTP статус (за замовчуванням 200). |
| `assertions.contains_any` | Проходить, якщо відповідь містить будь-який із цих підрядків. |
| `assertions.contains_all` | Проходить, якщо відповідь містить усі ці підрядки. |
| `assertions.contains_none` | Проходить, якщо відповідь не містить жодного з цих підрядків. |
| `save_response_id_as` | Зберегти ідентифікатор відповіді для наступного багатоетапного кроку. |
| `use_previous_response_id` | Надіслати цей запит, зв’язаний з раніше збереженим ідентифікатором відповіді. |

Перевірки є нечутливими до регістру перевірками наявності підрядка. Докладніше дивіться у
[документації дії](https://github.com/marketplace/actions/ai-smoke-test) щодо
повної схеми, включаючи керовані Foundry ресурси розмови.
