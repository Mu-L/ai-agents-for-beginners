# ⚡ Конкурентні робочі процеси агентів з Azure OpenAI (Responses API) (.NET)

## 📋 Навчальний посібник з високопродуктивної паралельної обробки

У цьому зошиті демонструються **патерни конкурентних робочих процесів** із використанням Microsoft Agent Framework для .NET та Azure OpenAI (Responses API). Ви дізнаєтесь, як створювати високопродуктивні робочі процеси паралельної обробки, які максимізують пропускну здатність, виконуючи кілька агентів ШІ одночасно, зберігаючи при цьому координацію та узгодженість даних.

## 🎯 Цілі навчання

### 🚀 **Основи конкурентної обробки**
- **Паралельне виконання агентів**: запуск кількох агентів ШІ одночасно для максимальної продуктивності
- **Патерни Async/Await**: використання моделі асинхронного програмування .NET для ефективної конкуренції
- **Azure OpenAI (Responses API)**: координація кількох конкурентних викликів до Responses API Azure OpenAI
- **Управління ресурсами**: ефективне керування ресурсами моделей ШІ у конкурентних операціях

### 🏗️ **Розширена архітектура конкуренції**
- **Паралелізм на основі завдань**: використання Task Parallel Library .NET для оптимального конкурентного виконання
- **Патерни синхронізації**: координація конкурентних агентів з уникненням гонок
- **Балансування навантаження**: ефективний розподіл роботи між доступними конкурентними потужностями
- **Стійкість до збоїв**: обробка окремих збоїв агентів без зупинки всього робочого процесу

### 🏢 **Корпоративні конкурентні застосунки**
- **Обробка великих обсягів документів**: одночасна обробка кількох документів
- **Аналіз контенту в режимі реального часу**: конкурентний аналіз вхідних потоків даних
- **Оптимізація пакетної обробки**: максимізація пропускної здатності для масштабних обробок даних
- **Мультиформатний аналіз**: паралельна обробка різних типів і форматів контенту

## 💻 Запуск коду

Повна реалізація доступна в файлі `03.dotnet-agent-framework-workflow-ghmodel-concurrent.cs`. Цей файл демонструє **конкурентний робочий процес Fan-Out/Fan-In** для планування подорожей:

### 🏗️ **Архітектура робочого процесу**

```
User Request → ConcurrentStartExecutor → [Researcher Agent || Planner Agent] → ConcurrentAggregationExecutor → Final Output
```

**Ключові компоненти:**

1. **ConcurrentStartExecutor**: одночасно розсилає запит користувача всім агентам
2. **Researcher Agent**: одночасно аналізує напрямки та цікаві місця
3. **Planner Agent**: одночасно створює детальні плани подорожі
4. **ConcurrentAggregationExecutor**: збирає та об’єднує результати від обох агентів

### 🎯 **Патерн Fan-Out/Fan-In**

Цей робочий процес демонструє класичний патерн **Fan-Out/Fan-In**:
- **Fan-Out**: один вхідний запит надсилається одночасно кільком агентам
- **Конкурентна обробка**: кілька агентів працюють паралельно над одним завданням
- **Fan-In**: результати від усіх агентів збираються та агрегуються в один вихід

### 🔍 Розуміння кастомних виконавців

**ConcurrentStartExecutor:**
- Реалізує `IMessageHandler<string>` для прийому рядкового вводу
- Розсилає повідомлення всім підключеним агентам
- Надсилає `TurnToken` для запуску конкурентної обробки

**ConcurrentAggregationExecutor:**
- Реалізує `IMessageHandler<ChatMessage>` для прийому відповідей агентів
- Збирає повідомлення у потокобезпечний спосіб
- Виконує агрегацію після отримання усіх очікуваних відповідей
- Повертає кінцевий результат за допомогою `context.YieldOutputAsync()`

### ⚡ Переваги продуктивності

**Конкурентне проти послідовного:**
- Послідовне: Agent1 (30с) → Agent2 (30с) = **всього 60 секунд**
- Конкурентне: Agent1 (30с) || Agent2 (30с) = **всього 30 секунд**

**Покращення пропускної здатності**: до N разів швидше для N конкурентних агентів (залежно від навантаження та ресурсів)
