# ⚡ Samanaikaiset agenttityönkulut Azure OpenAI:n (Responses API) kanssa (.NET)

## 📋 Suorituskykyinen rinnakkaiskäsittelyopas

Tämä muistikirja esittelee **samanaikaiset työnkulkumallit** Microsoft Agent Frameworkin .NET-versiolla ja Azure OpenAI:lla (Responses API). Opit rakentamaan suorituskykyisiä, rinnakkaisesti toimivia työnkulkuja, jotka maksimoivat läpimenon suorittamalla useita tekoälyagentteja samanaikaisesti pitäen samalla koordinaation ja tietojen yhdenmukaisuuden.

## 🎯 Oppimistavoitteet

### 🚀 **Samanaikaisen käsittelyn perusteet**
- **Agenttien rinnakkainen suoritus**: Suorita useita tekoälyagentteja samanaikaisesti maksimaalisen suorituskyvyn saavuttamiseksi
- **Async/Await-mallit**: Hyödynnä .NET:n async-ohjelmointimallia tehokkaaseen samanaikaisuuteen
- **Azure OpenAI (Responses API)**: Koordinoi useita samanaikaisia kutsuja Azure OpenAI Responses API:in
- **Resurssien hallinta**: Hallitse tekoälymallin resursseja tehokkaasti samanaikaisissa operaatioissa

### 🏗️ **Kehittynyt samanaikaisuusarkkitehtuuri**
- **Tehtäväpohjainen rinnakkaisuus**: Käytä .NET Task Parallel Libraryä optimoituun samanaikaiseen suoritukseen
- **Synkronointimallit**: Koordinoi rinnakkaiset agentit estäen kilpailutilanteet
- **Kuormantasapaino**: Jaa työ kuormittavasti käytettävissä olevalle samanaikaisten prosessien kapasiteetille
- **Vikasietoisuus**: Käsittele yksittäiset agenttien virheet pysäyttämättä koko työnkulkua

### 🏢 **Yritysten samanaikaiset sovellukset**
- **Suuren volyymin asiakirjakäsittely**: Käsittele useita asiakirjoja samanaikaisesti
- **Reaaliaikainen sisällön analysointi**: Samanaikainen saapuvien tietovirtojen analyysi
- **Eräprosessoinnin optimointi**: Maksimoi läpimeno suurissa tietojenkäsittelyoperaatioissa
- **Monimuotoinen analyysi**: Eri sisältötyyppien ja -formaattien rinnakkainen käsittely

## 💻 Koodin suorittaminen

Kokonaisimplementaatio löytyy tiedostosta `03.dotnet-agent-framework-workflow-ghmodel-concurrent.cs`. Tämä tiedosto esittelee **Fan-Out/Fan-In samanaikaisen työnkulun** matkasuunnittelua varten:

### 🏗️ **Työnkulun arkkitehtuuri**

```
User Request → ConcurrentStartExecutor → [Researcher Agent || Planner Agent] → ConcurrentAggregationExecutor → Final Output
```

**Keskeiset komponentit:**

1. **ConcurrentStartExecutor**: Lähettää käyttäjän pyynnön kaikille agenteille samanaikaisesti
2. **Researcher Agent**: Analysoi kohteita ja nähtävyyksiä rinnakkain
3. **Planner Agent**: Luo yksityiskohtaiset matkasuunnitelmat samanaikaisesti
4. **ConcurrentAggregationExecutor**: Kerää ja yhdistää molempien agenttien tulokset

### 🎯 **Fan-Out/Fan-In -malli**

Tämä työnkulku demonstroi klassista **Fan-Out/Fan-In** mallia:
- **Fan-Out**: Yksi syöteviesti lähetetään samanaikaisesti useille agenteille
- **Samanaikainen käsittely**: Useat agentit työskentelevät rinnakkain saman tehtävän parissa
- **Fan-In**: Kaikkien agenttien tulokset kerätään ja yhdistetään yhdeksi vastaukseksi

### 🔍 Mukautettujen suorittajien ymmärtäminen

**ConcurrentStartExecutor:**
- Toteuttaa `IMessageHandler<string>` vastaanottaakseen merkkijonosyötteen
- Lähettää viestit kaikille yhdistetyille agenteille
- Lähettää `TurnToken`:in käynnistääkseen samanaikaisen käsittelyn

**ConcurrentAggregationExecutor:**
- Toteuttaa `IMessageHandler<ChatMessage>` vastaanottaakseen agenttien vastauksia
- Kerää viestit säikeiturvallisesti
- Yhdistää vastaukset, kun kaikki odotetut vastaukset saapuvat
- Tuottaa lopputuloksen käyttäen `context.YieldOutputAsync()`

### ⚡ Suorituskyvyn hyödyt

**Samanaikainen vs peräkkäinen:**
- Peräkkäinen: Agentti1 (30s) → Agentti2 (30s) = **60 sekuntia yhteensä**
- Samanaikainen: Agentti1 (30s) || Agentti2 (30s) = **30 sekuntia yhteensä**

**Läpimenotehokkuuden parannus**: Jopa N-kertainen nopeutus N samanaikaiselle agentille (kuormituksen ja resurssien mukaan)
