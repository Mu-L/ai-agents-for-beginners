# ⚡ Samanaikaiset agenttityönkulut Azure OpenAI:n kanssa (Responses API) (.NET)

## 📋 Korkean suorituskyvyn rinnakkaiskäsittelyopas

Tämä muistikirja esittelee **samanaikaiset työnkulkujen mallit** käyttäen Microsoft Agent Frameworkia .NET:lle ja Azure OpenAI:ta (Responses API). Opit rakentamaan korkean suorituskyvyn rinnakkaiskäsittelytyönkulkuja, jotka maksimoivat läpimenon suorittamalla useita tekoälyagentteja samanaikaisesti säilyttäen samalla koordinoinnin ja tietojen johdonmukaisuuden.

## 🎯 Oppimistavoitteet

### 🚀 **Samanaikaisen käsittelyn perusteet**
- ** rinnakkainen agenttien suoritus**: Suorita useita tekoälyagentteja samanaikaisesti maksimaalisen suorituskyvyn saavuttamiseksi
- **Async/Await-mallit**: Hyödynnä .NET:n asynkronista ohjelmointimallia tehokkaaseen samanaikaisuuteen
- **Azure OpenAI (Responses API)**: Koordinoi useita samanaikaisia kutsuja Azure OpenAI Responses API:lle
- **Resurssien hallinta**: Hallitse tehokkaasti tekoälymallin resursseja samanaikaisissa toiminnoissa

### 🏗️ **Edistynyt samanaikaisuusarkkitehtuuri**
- **Tehtäväpohjainen rinnakkaisuus**: Käytä .NET Task Parallel Libraryä optimaaliseen samanaikaiseen suorittamiseen
- **Synkronointimallit**: Koordinoi samanaikaiset agentit estäen kilpajuoksuehtoja
- **Kuormantasapaino**: Jaa työkuormaa tehokkaasti käytettävissä olevan rinnakkaiskäsittelykapasiteetin kesken
- **Vikatoleranssi**: Käsittele yksittäiset agenttien virheet pysäyttämättä koko työnkulkua

### 🏢 **Yrityksen samanaikaiset sovellukset**
- **Suuren määrän asiakirjojen käsittely**: Käsittele useita asiakirjoja samanaikaisesti
- **Reaaliaikainen sisällön analyysi**: Saapuvien tietovirtojen samanaikainen analysointi
- **Eräprosessoinnin optimointi**: Maksimoi läpimeno suurissa tietojenkäsittelytoiminnoissa
- **Monimuotoinen analyysi**: Rinnakkainen käsittely erilaisille sisältötyypeille ja -muodoille

## 💻 Koodin suorittaminen

Täydellinen toteutus löytyy tiedostosta `03.dotnet-agent-framework-workflow-ghmodel-concurrent.cs`. Tämä tiedosto osoittaa **Fan-Out/Fan-In samanaikaisen työnkulun** matkasuunnitteluun:

### 🏗️ **Työnkulun arkkitehtuuri**

```
User Request → ConcurrentStartExecutor → [Researcher Agent || Planner Agent] → ConcurrentAggregationExecutor → Final Output
```

**Keskeiset osat:**

1. **ConcurrentStartExecutor**: Lähettää käyttäjän pyynnön kaikille agenteille samanaikaisesti
2. **Researcher Agent**: Analysoi matkakohteita ja nähtävyyksiä rinnakkain
3. **Planner Agent**: Luo yksityiskohtaiset matka- ja logistiset suunnitelmat rinnakkain
4. **ConcurrentAggregationExecutor**: Kerää ja yhdistää molempien agenttien tulokset

### 🎯 **Fan-Out/Fan-In -malli**

Tämä työnkulku esittää klassisen **Fan-Out/Fan-In** -mallin:
- **Fan-Out**: Yksi syöteviesti lähetetään samanaikaisesti useille agenteille
- **Samanaikainen käsittely**: Useat agentit työskentelevät rinnakkain saman tehtävän parissa
- **Fan-In**: Kaikkien agenttien vastaukset kerätään ja yhdistetään yhdeksi lopputulokseksi

### 🔍 Mukautettujen suorittajien ymmärtäminen

**ConcurrentStartExecutor:**
- Toteuttaa `IMessageHandler<string>` hyväksyäkseen merkkijonosyötteen
- Lähettää viestejä kaikille yhteyksissä oleville agenteille
- Lähettää `TurnToken` laukaistakseen samanaikaisen käsittelyn

**ConcurrentAggregationExecutor:**
- Toteuttaa `IMessageHandler<ChatMessage>` vastaanottaakseen agenttien vastaukset
- Kerää viestejä säikeiturvallisesti
- Yhdistää ne, kun kaikki odotetut vastaukset ovat tulleet
- Tuottaa lopullisen tuloksen käyttämällä `context.YieldOutputAsync()`

### ⚡ Suorituskyvyn edut

**Samanaikainen vs Peräkkäinen:**
- Peräkkäinen: Agentti1 (30s) → Agentti2 (30s) = **60 sekuntia yhteensä**
- Samanaikainen: Agentti1 (30s) || Agentti2 (30s) = **30 sekuntia yhteensä**

**Läpimenon parannus**: Jopa N-kertainen nopeutus N samanaikaisella agentilla (riippuen kuormituksesta ja resursseista)
