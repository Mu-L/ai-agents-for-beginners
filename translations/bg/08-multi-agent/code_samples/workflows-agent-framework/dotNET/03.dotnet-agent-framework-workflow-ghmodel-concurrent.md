# ⚡ Паралелни работни потоци на агенти с Azure OpenAI (Responses API) (.NET)

## 📋 Урок за високопроизводителна паралелна обработка

Този тетрадка демонстрира **паралелни модели на работни потоци** с използване на Microsoft Agent Framework за .NET и Azure OpenAI (Responses API). Ще научите как да изграждате високопроизводителни, паралелни работни потоци, които максимизират пропускателната способност чрез изпълнение на множество AI агенти едновременно, като същевременно поддържат координация и консистентност на данните.

## 💻 Стартиране на кода

Пълната имплементация е налична в `03.dotnet-agent-framework-workflow-ghmodel-concurrent.cs`. Този файл демонстрира **Fan-Out/Fan-In паралелен работен поток** за планиране на пътуване:

### 🏗️ **Архитектура на работния поток**

```
User Request → ConcurrentStartExecutor → [Researcher Agent || Planner Agent] → ConcurrentAggregationExecutor → Final Output
```

**Ключови компоненти:**

1. **ConcurrentStartExecutor**: Разпраща потребителската заявка към всички агенти едновременно
2. **Researcher Agent**: Паралелно анализира дестинации и забележителности
3. **Planner Agent**: Паралелно създава детайлни планове за пътуване
4. **ConcurrentAggregationExecutor**: Събира и обединява резултатите от двата агента

### 🎯 **Fan-Out/Fan-In модел**

Този работен поток илюстрира класическия **Fan-Out/Fan-In** модел:
- **Fan-Out**: Едно входящо съобщение се разпраща едновременно до множество агенти
- **Паралелна обработка**: Множество агенти работят паралелно по една и съща задача
- **Fan-In**: Резултатите от всички агенти се събират и агрегирани в един изход

### 📝 Очакван резултат

Работният поток ще:
1. **Изпрати запитване**: Изпраща "Планирай пътуване до Сиатъл през декември" към двата агента
2. **Паралелна обработка**: И двата агента работят едновременно:
   - Изследователят идентифицира забележителности и детайли
   - Планировчикът подготвя маршрут и логистика
3. **Агрегация**: Комбинира и двата отговора в окончателен обширен изход
4. **Показване на резултатите**: Показва обединения пътен план с цялата информация

Този паралелен модел е идеален за:
- **Създаване на съдържание**: Множество писатели създават различни секции едновременно
- **Преглед на код**: Множество рецензенти анализират код от различни гледни точки
- **Маркетингови проучвания**: Паралелен анализ на различни пазарни сегменти
- **Обработка на документи**: Паралелно извличане, анализ и валидиране
- **Многоаспектен анализ**: Получаване на различни гледни точки за един и същи вход

### 🔍 Разбиране на персонализирани изпълнители

**ConcurrentStartExecutor:**
- Имплементира `IMessageHandler<string>` за приемане на входящи низове
- Разпраща съобщения до всички свързани агенти
- Изпраща `TurnToken`, за да задейства паралелната обработка

**ConcurrentAggregationExecutor:**
- Имплементира `IMessageHandler<ChatMessage>` за приемане на отговори от агенти
- Събира съобщения по потокобезопасен начин
- Агрегира когато пристигнат всички очаквани отговори
- Връща крайния изход чрез `context.YieldOutputAsync()`

### ⚡ Предимства в производителността

**Паралелно срещу поредно:**
- Поредно: Агент1 (30с) → Агент2 (30с) = **общо 60 секунди**
- Паралелно: Агент1 (30с) || Агент2 (30с) = **общо 30 секунди**

**Подобрение на пропускателната способност**: До N× по-бързо за N паралелни агенти (в зависимост от натоварването и ресурсите)

### 🛡️ Обработка на грешки

Работният поток обработва индивидуални поражения на агенти деликатно:
- Ако един агент се провали, други продължават да работят
- Агрегаторът може да прилага timeout логика
- Могат да се върнат частични резултати при необходимост
