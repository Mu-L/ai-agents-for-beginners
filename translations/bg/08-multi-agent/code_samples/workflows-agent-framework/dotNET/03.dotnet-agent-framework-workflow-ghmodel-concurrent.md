# ⚡ Конкурентни работни потоци на агенти с Azure OpenAI (Responses API) (.NET)

## 📋 Урок за високоефективна паралелна обработка

Този тетрадка демонстрира **конкурентни модели на работни потоци** с използване на Microsoft Agent Framework за .NET и Azure OpenAI (Responses API). Ще научите как да изградите високоефективни работни потоци за паралелна обработка, които максимизират пропускателната способност чрез изпълнение на множество AI агенти едновременно при запазване на координацията и консистентността на данните.

## 💻 Стартиране на кода

Пълната имплементация е налична в `03.dotnet-agent-framework-workflow-ghmodel-concurrent.cs`. Този файл демонстрира **Fan-Out/Fan-In конкурентен работен поток** за планиране на пътуване:

### 🏗️ **Архитектура на работния поток**

```
User Request → ConcurrentStartExecutor → [Researcher Agent || Planner Agent] → ConcurrentAggregationExecutor → Final Output
```

**Основни компоненти:**

1. **ConcurrentStartExecutor**: Разпространява потребителската заявка към всички агенти едновременно
2. **Researcher Agent**: Анализира дестинации и атракции конкурентно
3. **Planner Agent**: Създава детайлни планове за пътуване конкурентно
4. **ConcurrentAggregationExecutor**: Събира и обединява резултатите и от двата агента

### 🎯 **Модел Fan-Out/Fan-In**

Този работен поток демонстрира класическия модел **Fan-Out/Fan-In**:
- **Fan-Out**: Един входящо съобщение се разпръсква към множество агенти едновременно
- **Конкурентна обработка**: Множество агенти работят паралелно по същата задача
- **Fan-In**: Резултатите от всички агенти се събират и агрегират в един изход

Този конкурентен модел е идеален за:

- **Преглед на код**: Множество ревюъри анализират кода от различни гледни точки
- **Пазарни изследвания**: Паралелен анализ на различни пазарни сегменти
- **Обработка на документи**: Конкурентното извличане, анализ и валидиране
- **Анализ от множество перспективи**: Получаване на разнообразни гледни точки върху един и същи вход

### 🔍 Разбиране на персонализираните изпълнители

**ConcurrentStartExecutor:**
- Имплементира `IMessageHandler<string>` за приемане на входен низ
- Разпространява съобщения до всички свързани агенти
- Изпраща `TurnToken`, за да задейства конкурентната обработка

**ConcurrentAggregationExecutor:**
- Имплементира `IMessageHandler<ChatMessage>` за приемане на отговори от агенти
- Събира съобщения по нишково-безопасен начин
- Агрегира, когато пристигнат всички очаквани отговори
- Връща окончателния изход чрез `context.YieldOutputAsync()`

### ⚡ Ползи от производителността

**Конкурентно срещу последователно:**
- Последователно: Agent1 (30s) → Agent2 (30s) = **общо 60 секунди**
- Конкурентно: Agent1 (30s) || Agent2 (30s) = **общо 30 секунди**

**Подобрение на пропускателната способност**: До N× по-бързо за N конкуретни агенти (в зависимост от натоварването и ресурсите)

### 🛡️ Обработка на грешки

Работният поток обработва индивидуални грешки на агенти грациозно:

- Агрегаторът може да реализира логика за таймаут
- При нужда могат да се върнат частични резултати
