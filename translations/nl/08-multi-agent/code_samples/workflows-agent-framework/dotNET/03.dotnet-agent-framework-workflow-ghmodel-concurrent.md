## 💻 De Code Uitvoeren

De volledige implementatie is beschikbaar in `03.dotnet-agent-framework-workflow-ghmodel-concurrent.cs`. Dit bestand demonstreert een **Fan-Out/Fan-In gelijktijdige workflow** voor reisplanning:

### 🏗️ **Workflow Architectuur**

```
User Request → ConcurrentStartExecutor → [Researcher Agent || Planner Agent] → ConcurrentAggregationExecutor → Final Output
```

**Belangrijke componenten:**

1. **ConcurrentStartExecutor**: Zendt het gebruikersverzoek simultaan naar alle agents
2. **Researcher Agent**: Analyseert bestemmingen en attracties gelijktijdig
3. **Planner Agent**: Maakt gedetailleerde reisplannen gelijktijdig
4. **ConcurrentAggregationExecutor**: Verzamelt en voegt resultaten van beide agents samen

### 🎯 **Fan-Out/Fan-In Patroon**

Deze workflow demonstreert het klassieke **Fan-Out/Fan-In** patroon:
- **Fan-Out**: Eén inputbericht wordt simultaan uitgezonden naar meerdere agents
- **Gelijktijdige Verwerking**: Meerdere agents werken parallel aan dezelfde taak
- **Fan-In**: Resultaten van alle agents worden verzameld en samengevoegd tot één uitvoer

### 🔍 Begrijp Custom Executors

**ConcurrentStartExecutor:**
- Implementeert `IMessageHandler<string>` om string input te accepteren
- Zendt berichten uit naar alle verbonden agents
- Stuurt `TurnToken` om gelijktijdige verwerking te activeren

**ConcurrentAggregationExecutor:**
- Implementeert `IMessageHandler<ChatMessage>` om agentantwoorden te ontvangen
- Verzamelt berichten op een thread-veilige manier
- Voert aggregatie uit wanneer alle verwachte antwoorden binnen zijn
- Genereert finale output met `context.YieldOutputAsync()`

### ⚡ Prestatievoordelen

**Gelijktijdig vs Sequentieel:**
- Sequentieel: Agent1 (30s) → Agent2 (30s) = **60 seconden totaal**
- Gelijktijdig: Agent1 (30s) || Agent2 (30s) = **30 seconden totaal**

**Doorvoerverbetering**: Tot N× sneller voor N gelijktijdige agents (afhankelijk van werklast en middelen)

### 🛡️ Foutafhandeling

De workflow gaat soepel om met individuele agentstoringen:
- Als één agent faalt, blijven de anderen verwerken
- Aggregator kan time-out-logica implementeren
- Gedeeltelijke resultaten kunnen worden teruggegeven indien nodig
